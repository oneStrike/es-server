# Task / Check-In 执行排期

本文件是本工作包唯一的排期事实源，负责维护：

- 优先级；
- 执行波次；
- 依赖关系；
- 任务状态；
- 变更记录。

除本文件外，其他文档只允许引用排期结论，不允许维护第二套排序或依赖。

## 优先级说明

- `S0`：必须先完成的基础设计与数据契约。
- `S1`：主链路接口与读模型建设。
- `S2`：对账、补偿、上线验收与运行治理。

## 当前默认决策

1. 签到一期主链路完全独立于 `task`。
2. 签到一期奖励结算完全独立于现存 `DAILY_CHECK_IN` 事件配置。
3. `GrowthRuleTypeEnum.DAILY_CHECK_IN = 6` 保留为预留编码，但不参与当前可配置事件视图与一期结算主链路。
4. 第一阶段只允许一个当前有效签到计划，避免多计划并行带来的周期、补签额度和奖励归属歧义。
5. 基础签到奖励由 `check_in_plan.baseRewardConfig` 提供，连续签到奖励由 `check_in_streak_reward_rule` 提供，二者都直接进入成长账本。

## 依赖定义

- `硬前置`：未完成前不得开工。
- `软前置`：建议先完成，但允许在契约已冻结后并行。
- `可并行`：允许同时推进，但必须满足并行前提。
- `直接后置`：本任务完成后最先承接的任务。

## 波次划分

### Wave 1

- `P0-01` Task 全量梳理与签到边界确认
- `P0-02` 签到计划、周期与补签契约
- `P0-03` 签到记录、基础奖励与连续奖励结算

### Wave 2

- `P1-01` App/Admin 签到接口与读模型

### Wave 3

- `P2-01` 对账、补偿与上线验收

## 并行策略

1. `P0-02` 与 `P0-03` 只允许在以下契约冻结后并行：
   - `check_in_plan` 字段模型；
   - 周期切片规则；
   - 补签额度扣减规则；
   - 基础奖励与连续奖励的幂等键。
2. `P1-01` 必须等待 `P0-02`、`P0-03` 全部完成，因为接口读写都依赖最终表结构与结算语义。
3. `P2-01` 不与主实现并行上线，必须在接口和结算链路稳定后再做最终收口。

## 资源冲突约束

1. `P0-02` 与 `P0-03` 都会触碰签到表结构设计，必须以 `P0-02` 冻结的字段契约为准。
2. `P0-03` 与 `P1-01` 都会使用成长账本能力，但 `P1-01` 不得反向定义奖励语义。
3. 本工作包不允许在任何任务中重新引入 `task` 依赖或现存 `DAILY_CHECK_IN` 配置依赖；若确有新需求，必须新增独立工作包。

## 里程碑

1. `M1`：完成 `P0-01` 到 `P0-03`，冻结签到数据模型、奖励模型和补签规则。
2. `M2`：完成 `P1-01`，打通 App/Admin 的配置、签到、补签、日历与记录视图。
3. `M3`：完成 `P2-01`，补齐对账、补偿、运行观察与最终验收证据。
