# 最终成果验收清单

## 1. 文档目标

本文只保留“跨任务、跨模块”的统一验收项。

它不是：

- 阶段计划
- 开发排期版
- 单任务完成标准汇总表

使用方式：

1. 各任务自己的完成标准，以对应 work-item 文件为准
2. 联调、回归、准备上线时，再按本文核对跨任务事项
3. 每一项都要有代码、接口、数据、测试或文档证据

## 2. 单任务验收事实源

每个任务自己的目标、范围、完成标准、非目标、代码锚点，以对应 task 文件为准：

- `docs/task-growth-reward-work-items/p0/*`
- `docs/task-growth-reward-work-items/p1/*`
- `docs/task-growth-reward-work-items/p2a/*`
- `docs/task-growth-reward-work-items/p2b/*`
- `docs/task-growth-reward-work-items/p2c/*`

本文不再重复抄写各任务的细项完成标准，避免多份清单漂移。

## 3. 总体验收结论标准

以下条件全部满足，才能认定“最终成果验收通过”：

- [ ] 奖励时机、治理口径、规则编码三者一致
- [ ] 人工补发、举报裁决、主题审核补发都具备稳定幂等
- [ ] 任务奖励从任务侧、账本侧、接口侧都可解释
- [ ] App 与管理端都能解释成长账本来源
- [ ] 事件定义有唯一事实源，旧编码和旧注释不再扩散
- [ ] 通知域具备模板、偏好、delivery、提醒、公告边界的最小闭环
- [ ] 治理层真正成为奖励、通知、任务推进的正式闸门
- [ ] 核心链路有自动化测试覆盖，且主要测试通过
- [ ] 文档、代码、seed、DTO 口径一致

## 4. 数据与迁移验收

- [ ] 所有新增表、字段、索引均有对应 migration
- [ ] migration 在空库可执行
- [ ] migration 在已有数据环境可执行
- [ ] 新字段默认值或回填策略明确
- [ ] 不会因为新增字段导致老数据查询报错
- [ ] schema 与 relations 已同步更新

验收证据：

- [ ] migration 执行日志
- [ ] schema 与 relations 抽查结果
- [ ] 至少做过一次本地从迁移到接口验证的完整回归

## 5. 接口与兼容性验收

- [ ] 新增接口已补 DTO
- [ ] 新增接口已补说明注释
- [ ] 旧接口若保留，行为未被静默破坏
- [ ] 旧接口若废弃，文档中有明确迁移说明
- [ ] App 与管理端都能拿到新增关键字段
- [ ] 兼容性变化不会让旧前端在无感情况下崩掉

验收证据：

- [ ] swagger 或 DTO 可证明字段存在
- [ ] 老接口 smoke test 通过

## 6. 可观测性与幂等验收

- [ ] 人工补发链路能用稳定键串起账本、日志、审计
- [ ] 奖励主链路失败时能区分“规则缺失 / 限额 / 幂等命中 / 真实异常”
- [ ] 通知域能区分“投递成功 / 失败 / 重试 / 主动跳过”
- [ ] 治理闸门前后的行为差异可从日志或数据中解释

验收证据：

- [ ] 至少抽查人工补发、举报裁决、任务奖励、通知投递各一条完整链路

## 7. 测试与质量验收

### 7.1 单元与集成测试

- [ ] 核心任务都有对应自动化测试或明确回归记录
- [ ] 主题审核补发、举报裁决发奖、任务奖励状态回写至少覆盖一条自动化路径
- [ ] 通知模板、偏好、delivery 若落地，至少各有基础测试
- [ ] 评论审核后台与 chat outbox 若落地，也有基础回归测试

### 7.2 命令验收

- [ ] `pnpm type-check` 通过
- [ ] `pnpm exec tsc -p tsconfig.json --noEmit` 通过
- [ ] 相关测试命令通过
- [ ] 新增 spec 不依赖脆弱的时间戳或随机顺序

### 7.3 回归重点

- [ ] 发帖创建主链路未回归
- [ ] 举报创建主链路未回归
- [ ] 任务领取、进度上报、手动完成未回归
- [ ] App 通知分页、未读数、已读操作未回归
- [ ] 管理端成长、任务、消息模块基础功能未回归

## 8. 文档与交付验收

- [ ] 领域设计文档已更新
- [ ] 阶段总览文档已更新
- [ ] 小任务文档已更新
- [ ] 执行计划文档已更新
- [ ] 开发排期版已更新
- [ ] 最终成果与文档描述一致
- [ ] 文档中不存在已知过期口径
- [ ] 文档不再引用仓内不存在的前端文件或页面

验收证据：

- [ ] 抽查领域文档、任务文档、接口 DTO，口径一致
- [ ] 新成员只看文档也能理解正确链路

## 9. 阻塞上线项

以下任一项未通过，都建议视为阻塞上线：

- [ ] 举报仍然是“创建即发奖”
- [ ] 主题待审核通过后仍无法补发 `CREATE_TOPIC`
- [ ] 人工补发仍然没有稳定 `operationKey`
- [ ] `rewardConfig` 仍允许录入当前不支持的奖励字段
- [ ] `task_assignment` 仍无法解释任务奖励状态
- [ ] App / 管理端仍无法看到账本来源字段
- [ ] 规则编码、seed、注释仍然明显冲突
- [ ] 通知域新增表和链路缺少最基本测试

## 10. 最终签收标准

当以下问题都能回答“是”时，可以签收：

- [ ] 我们能否准确解释任一奖励为什么发、什么时候发、为什么没发？
- [ ] 我们能否稳定重试人工补发而不重复记账？
- [ ] 我们能否从任务记录直接判断奖励结算状态？
- [ ] 我们能否从账本接口直接判断来源，而不是依赖猜测？
- [ ] 我们能否明确指出事件定义的唯一事实源？
- [ ] 我们能否说清楚通知、公告、提醒、chat 的边界？
- [ ] 我们能否证明治理结果已经成为正式闸门？
- [ ] 我们能否让新同事只看文档就不被旧口径误导？

只要以上问题全部回答“是”，这轮任务才算真正完成。
