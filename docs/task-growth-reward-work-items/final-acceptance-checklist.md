# 最终成果验收清单

## 1. 文档目的

本文用于在全部任务完成后，对最终成果做统一验收。

它不是阶段计划，也不是开发任务单，而是最终交付时的核对清单。

建议使用方式：

1. 先完成对应任务开发
2. 再按本文逐项核对
3. 每一项都要有代码、接口、数据、测试或文档证据

## 2. 涉及模块

- `apps/admin-api`
- `apps/app-api`
- `libs/growth`
- `libs/interaction`
- `libs/forum`
- `libs/message`
- `libs/app-content`
- `db/schema`
- `db/seed`
- `docs`

## 3. 总体验收结论标准

以下条件全部满足，才能认定“最终成果验收通过”：

- [ ] 奖励时机、治理口径、规则编码三者一致
- [ ] 人工补发、举报裁决、主题审核补发都具备稳定幂等
- [ ] 任务奖励从任务侧、账本侧、接口侧都可解释
- [ ] App 与管理端都能解释成长账本来源
- [ ] 事件定义有唯一事实源，旧编码和旧注释不再扩散
- [ ] 通知域具备模板、偏好、delivery、提醒、公告边界的最小闭环
- [ ] 治理层真正成为奖励、通知、任务推进的正式闸门
- [ ] 核心链路有自动化测试覆盖，且主要测试通过
- [ ] 文档、代码、seed、DTO 口径一致

## 14. 最终签收标准

当以下问题都能回答“是”时，可以签收：

- [ ] 我们能否准确解释任一奖励为什么发、什么时候发、为什么没发？
- [ ] 我们能否稳定重试人工补发而不重复记账？
- [ ] 我们能否从任务记录直接判断奖励结算状态？
- [ ] 我们能否从账本接口直接判断来源，而不是依赖猜测？
- [ ] 我们能否明确指出事件定义的唯一事实源？
- [ ] 我们能否说清楚通知、公告、提醒、chat 的边界？
- [ ] 我们能否证明治理结果已经成为正式闸门？
- [ ] 我们能否让新同事只看文档就不被旧口径误导？

只要以上问题全部回答“是”，这轮任务才算真正完成。
