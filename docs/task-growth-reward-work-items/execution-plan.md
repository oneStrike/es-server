# 任务执行计划

## 1. 文档目标

本文用于把拆分后的任务单再进一步变成“可排期、可并行、可交付”的执行计划。

它解决 4 个问题：

1. 哪些任务优先级最高
2. 哪些任务有硬前置依赖
3. 哪些任务可以并行推进
4. 建议按什么波次实施

如需查看每个任务对应的预计改动模块、关键文件和测试点，请配合阅读：

- [开发排期版](./development-plan.md)

## 2. 优先级定义

- `S0`：立即执行。直接影响正确性、幂等、治理口径，属于 blocker
- `S1`：高优先。直接影响主链路解释力或交付完整度
- `S2`：中优先。属于下一阶段关键建设，但不是当前 blocker
- `S3`：后置优先。价值明确，但应放在更靠后阶段推进

## 3. 依赖说明

- `硬前置`：不完成前通常不建议开工，否则会返工或语义不稳
- `软前置`：不是绝对 blocker，但建议先完成，能减少回改成本
- `直接后置`：该任务完成后最直接解锁的后续任务
- `可并行`：写入面和语义边界相对独立，可并行推进

## 6. 并行原则

建议只有在以下条件同时成立时再并行：

1. 没有硬前置依赖
2. 改动文件集合基本独立
3. 不会同时改同一套 DTO、同一组 schema、同一条主链路的时机口径

不建议并行的典型组合：

- `P0-02` 与 `P0-03` 同时落地
- `P1-02` 与 `P1-03` 反向并行
- `P2-A` 与 `P2-B` 同轮启动
- `P2-B-04` 与 `P2-C-01` 同时进入实施阶段

## 7. 最终建议

这批任务最稳妥的推进方式是：

`先修正确性与幂等 -> 再补任务与账本解释力 -> 再建设定义层 -> 最后补通知与治理闭环`

只要顺序不反，任务会越做越清晰；一旦顺序反了，就会重新回到“大方案先行、小问题悬空”的状态。
