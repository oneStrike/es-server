# 种子数据真实化改造设计

> 创建日期：2026-07-21
> 状态：已批准，待实现

## 背景与目标

当前 `db/seed/` 下的种子数据虽然覆盖了全部业务域，但数据明显是模拟数据——用户昵称如 `tsuki-note` 的拼接痕迹、作品仅 4 部、互动数据规模小。目标是将其改造为**看起来像真实运营了一个季度的平台**的数据集，具体要求：

1. 使用真实的漫画/小说作品名和作者名
2. 用户昵称像真实网名
3. 数据之间的关联正确（字典引用、计数器、外键、时间线）
4. 数据规模扩大到像一个运营了三个月的平台
5. 覆盖全部 6 个域：系统、管理员、应用用户、作品、论坛、消息

## 方案选择

采用**方案 C：混合模式**——核心实体（作品/作者/分类/标签/板块）硬编码真实数据，用户和互动数据程序化生成。

## 设计原则

- **先查业务代码再定义数据**：每个字段都要先查业务代码确认它是否由数据字典驱动。如果是字典驱动，往字典里加项；如果是枚举，用枚举值。
- **确定性生成**：使用固定 seed 值的伪随机生成器，保证每次运行结果一致（幂等性）
- **计数器一致性**：所有计数器字段（viewCount、favoriteCount 等）必须等于实际关联记录数
- **时间线合理性**：数据分布在 3 个月内，作品发布 → 用户注册 → 互动的因果顺序正确

### 3. 应用用户域（程序化生成 + 真实风格）

**用户规模：从 96 个扩大到 150 个。**

**用户昵称池：** 人工编写 150 个真实风格中文网名池，例如：

- "月下独酌"、"墨染樱前"、"深夜档追番人"、"夏日柠檬茶"
- "星尘与银河"、"旧书店的猫"、"雨夜调频"、"午后的红茶"
- "追番不设上限"、"二次元住民"、"漫画仓库管理员" 等

**用户行为画像分层：**

| 层级     | 数量 | 注册时间    | 特征                                   |
| -------- | ---- | ----------- | -------------------------------------- |
| 资深用户 | 20   | 3 个月前    | 经验值高，大量互动、购买记录、会员订阅 |
| 活跃用户 | 80   | 2-3 个月前  | 中等互动量                             |
| 新用户   | 50   | 最近 2-3 周 | 互动量少，部分只有浏览记录             |

**每个用户包含：**

- 真实风格昵称（从昵称池取）
- 合理的手机号、邮箱格式
- 真实城市/ISP 分布（北京/上海/广州/深圳/成都/杭州等，移动/联通/电信）
- `levelId` 引用真实存在的等级规则
- 合理的 `points`/`experience` 资产余额
- 部分用户有会员订阅记录
- 部分用户有禁言状态（`status=2`，带 `banReason` 和 `banUntil`）

**签到与成长数据：** 资深用户有连续签到记录，成长审计日志和积分流水。

**关键关联保证：**

- `levelId` 引用真实存在的等级规则
- 资产余额与 `GrowthAssetTypeEnum` 枚举一致
- 会员订阅引用真实存在的 `membershipPlan`

### 4. 论坛域（板块 + 话题 + 互动）

**板块分组（从 4 个扩充到 5 个）：**

- 站务与新人、番剧漫画讨论、创作与同人、周边与线下、作品评论区（新增）

**板块（从 8 个扩充到 12-14 个）：**

- 保留现有 8 个板块
- 新增：追番日程表、原作 vs 动画、设定考察党、补番互助
- 每个作品自动绑定一个专属讨论板块

**话题（半硬编码 + 程序化）：**

- 硬编码 20-30 个真实风格话题，例如：
  - "进击的巨人最终季观感汇总"
  - "咒术回战涉谷事变完全解读"
  - "葬送的芙莉莲为什么是神作"
  - "追番推荐：2026 春季新番一览"
- 每个话题有真实风格的内容（2-3 段文字），关联正确的板块和用户
- 程序化生成 50-80 个补充话题

**评论与回复：**

- 每个硬编码话题下有 5-15 条评论，评论内容是真实风格的讨论语句
- 评论有嵌套回复（2-3 层楼中楼）
- 评论时间线合理（话题发布后陆续评论）

**互动数据（点赞/收藏/浏览）：**

- 根据用户行为画像分层：资深用户互动多，新用户互动少
- 浏览记录 > 点赞 > 收藏（符合真实漏斗比例）
- 时间线分布在 3 个月内

**版主与审核：**

- 2-3 个版主（从资深用户中选），绑定到对应板块
- 少量审核操作日志

**作品互动数据（跨域关联）：**

| 数据类型 | 说明                             | 枚举引用                  |

| 阅读状态 | 资深用户有 5-10 个作品的阅读记录 | -                         |
| 章节购买 | 约 30% 的资深/活跃用户有购买记录 | `PurchaseTargetTypeEnum`  |
| 下载记录 | 部分用户有章节下载记录           | `DownloadTargetTypeEnum`  |
| 收藏     | 用户收藏作品                     | `FavoriteTargetTypeEnum`  |
| 点赞     | 用户点赞作品/章节/评论           | `LikeTargetTypeEnum`      |
| 评论     | 用户评论作品/章节                | `CommentTargetTypeEnum`   |

**消息域：**

- 通知模板：保留现有 10 个模板，全部从 `getCanonicalNotificationTemplateContract` 生成
- 聊天会话：3-5 个会话，每会话 5-15 条消息，消息内容是真实风格对话
- 站内通知：根据互动自动生成，`categoryKey` 与通知模板匹配
- WS 指标：保留现有数据

**系统运行数据：**

- 请求日志：从 2 条扩充到 15-20 条，覆盖 admin 和 app 端常见操作，`apiType` 引用 `ApiTypeEnum`

## 架构与文件组织

**文件组织原则：**

- 不新建 `fixtures/` 目录——保持当前 `db/seed/modules/{domain}/domain.ts` 的单文件结构
- 每个 domain 文件内的 fixture 数据以 `const` 数组定义在文件顶部
- `shared.ts` 扩充：增加新的时间线常量、扩充的字典 code/item 常量、昵称池

**改动文件清单：**

| 文件                                | 改动内容                                   |
| ----------------------------------- | ------------------------------------------ |
| `db/seed/shared.ts`                 | 扩充字典常量、时间线基准、昵称池           |
| `db/seed/modules/system/domain.ts`  | 扩充字典项、敏感词、请求日志               |
| `db/seed/modules/admin/domain.ts`   | 管理员信息微调                             |
| `db/seed/modules/work/domain.ts`    | 重写作品/作者/分类/标签 fixture 为真实数据 |
| `db/seed/modules/app/domain.ts`     | 重写用户 fixture、扩充互动数据生成         |
| `db/seed/modules/forum/domain.ts`   | 重写板块/话题/评论 fixture 为真实风格内容  |
| `db/seed/modules/message/domain.ts` | 扩充聊天/通知数据                          |

**执行顺序不变：**

1. 全局参考数据（系统字典 → 管理员 → 应用核心 → 论坛参考）
2. 论坛主体与互动数据
3. 系统运行数据

**幂等性保证：**

- 所有 seed 操作保持现有的 upsert 模式（先查再 insert/update）
- 确定性生成：使用固定 seed 值的伪随机生成器
- 清理逻辑保持现有的 `cleanupRetiredDemoDomains` 流程

## 验证策略

- `pnpm type-check` 确认类型安全
- 运行 seed 脚本确认无运行时错误
- 检查关键计数器一致性（可通过查询验证）
