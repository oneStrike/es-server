# Resolver 优化优先级清单（P0/P1/P2 + 预计收益）

## 1. 目标与范围

- 目标：给出全项目 Resolver 的可执行优化优先级清单，聚焦性能与可扩展性。
- 范围：`libs/interaction`、`libs/content`、`libs/forum`、`libs/growth` 中各类 Resolver 及其被 Service 调用模式。
- 口径：优先级按业务影响与实现成本综合评估。

## 2. 总览结论

- 当前最高风险点集中在 `reading-state` 链路，存在典型“列表聚合中的逐条 resolver 调用”模式。
- `like/favorite` 链路整体已采用 `batchGetDetails`，属于较优实现，主要是参数去重与容错增强。
- `report/comment/download/purchase/browse-log` 以单次业务动作调用为主，不是同类 N+1 主风险，但可做一致性与吞吐优化。

## 3. 按模块优先级清单

| 模块 | P0（本周优先） | P1（本迭代） | P2（排期优化） | 预计收益 |
|---|---|---|---|---|
| interaction / reading-state | 1) 为章节快照补齐批量查询能力，替代列表中的逐条调用；2) `touchByChapter` 避免按 resolver 顺序试探 | 1) 分组后按原顺序构建返回，避免额外排序；2) 缺失 work 快照时不做分页补偿，直接返回该条快照记录（`work` 置为快照占位结构） | 1) 热点用户短 TTL 缓存 | 列表接口 P95 延迟常见下降 30%~60%，DB 往返显著减少，且分页返回条数稳定 |
| interaction / favorite | 1) 分组后 `ids` 去重再 `batchGetDetails` | 1) resolver 失败维度化日志（targetType、批次大小、耗时） | 1) 目标详情缓存（按类型小粒度） | 大页/高重复 ID 场景下减少 10%~25% 明细查询成本 |
| interaction / like | 1) 保持 `batchGetDetails`，统一所有 resolver 的去重策略 | 1) 列表详情缺失时打点观测；2) 批量接口超时降级策略 | 1) 热门目标详情缓存 | 点赞列表稳定性提升，异常批次可观测性增强 |
| interaction / comment | 1) 事务内调用链最短化，避免无效重复校验 | 1) `resolveMeta`/`ensureCanComment` 共享查询结果，减少重复读 | 1) 楼层分配热点目标限流与队列化策略 | 高并发评论场景下事务时间缩短，冲突重试下降 |
| interaction / report | 1) 统一 resolver 元信息查询最小字段集 | 1) 批量举报审核场景预留 `batchResolveMeta` 契约 | 1) 审核场景 owner 信息缓存 | 单次收益中等，未来批量审核场景收益高 |
| interaction / download | 1) 各 resolver 的目标校验与计数更新统一快速路径 | 1) 目标存在性校验抽象公共 helper | 1) 高频章节计数异步合并写 | 写入链路 RT 稳定，热点更新压力下降 |
| interaction / purchase | 1) 与 download 同步统一 resolver 校验路径 | 1) 购买后计数更新与明细读取链路压缩 | 1) 针对大促场景的批量化预案 | 高并发购买峰值下吞吐更稳 |
| interaction / browse-log | 1) resolver 侧 `ensureTargetValid/applyCountDelta` 统一受影响行语义 | 1) post process 失败结构化日志 | 1) 浏览计数增量异步聚合 | 高流量下写放大下降，日志可观测性提升 |
| content / work & chapter resolvers | 1) 与 interaction 契约统一（批量接口命名、入参去重、最小字段） | 1) Prisma/Drizzle 混用 resolver 的查询风格统一 | 1) 章节/作品详情缓存抽象 | 降低跨模块维护成本，减少重复 SQL 模板 |
| forum / topic resolvers | 1) favorite/like/comment resolver 与 interaction 批量契约对齐 | 1) 历史混合风格 resolver 统一查询入口 | 1) 主题详情缓存与失效策略 | 列表类接口抖动下降，跨团队协作成本降低 |
| growth / user-report resolver | 1) 目标存在性校验语义与 interaction 统一 | 1) 审计字段标准化 | 1) 风控链路批量化预留 | 单次性能收益有限，但一致性与可维护性提升明显 |

## 4. 实施批次建议

### 批次 A（P0，1~2 周）

1. reading-state 批量章节快照 + `touchByChapter` 路由改造。
2. favorite/like 全链路去重标准化。
3. download/purchase/browse-log 的 resolver 受影响行语义统一。

### 批次 B（P1，2~3 周）

1. comment/report 的 resolver 查询复用与事务链路压缩。
2. content/forum resolver 契约统一。
3. 关键失败日志与指标打点补齐。

### 批次 C（P2，持续优化）

1. 热点详情缓存与异步聚合写。
2. 批量场景预留接口（`batchResolveMeta` 等）。
3. 结合线上画像做针对性调优。

## 5. 验收指标（建议）

- P0 完成后：
  - `reading-history` 接口：P95 延迟下降 ≥ 30%，DB 查询次数下降 ≥ 40%。
  - favorite/like 列表：同页重复 ID 查询量下降 ≥ 50%。
  - download/purchase/browse-log：关键写操作失败语义一致且可观测。
- P1 完成后：
  - comment/report 关键链路事务时长下降 ≥ 15%。
  - resolver 失败日志具备可聚合维度（type、batchSize、耗时、errorCode）。
- P2 完成后：
  - 热点目标查询命中缓存比例达到预期阈值（按业务设定）。

## 6. 当前建议优先落地点

1. 先做 `reading-state`（收益最高、改动面可控）。
2. 再做 favorite/like 去重标准化（低风险快收益）。
3. 最后并行推进 download/purchase/browse-log 的一致性重构。
