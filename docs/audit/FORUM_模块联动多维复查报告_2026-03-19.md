# Forum 模块联动多维复查报告

- 审查时间：2026-03-19
- 审查方式：静态代码审查 + 规范对照 + `pnpm type-check`
- 审查范围：
  - `db/schema/forum`
  - `db/relations/forum.ts`
  - `libs/forum/src`
  - `apps/admin-api/src/modules/forum`
  - `apps/app-api/src/modules/forum`
  - 论坛联动侧重点：`apps/app-api/src/modules/user`、`libs/user/src`、`libs/content/src/work`、`libs/interaction`
- 对照规范：
  - `.trae/rules/DTO_SPEC.md`
  - `.trae/rules/drizzle-guidelines.md`
  - `.trae/rules/project_rules.md`

## 总体结论

本轮复查确认，上一轮作品/论坛联动里最核心的几处断点已经补上了：

- `app/forum/topic` 已补齐基础 CRUD 入口。
- 论坛主题删除已限定 `targetType = FORUM_TOPIC`，未复现“按同 ID 误删其他目标评论”的旧问题。
- 论坛主题评论/点赞/收藏/浏览/举报 resolver 已统一补上“审核通过 + 非隐藏 + 板块启用”的公开可见性约束。
- `syncTopicCommentState()` / `syncSectionVisibleState()` 已形成主题与板块的可见计数闭环。
- `pnpm type-check` 当前通过。

但论坛模块仍然存在一批“看起来能跑、但契约和联动并不可靠”的问题，主要集中在五类：

- DTO 和表结构、真实回参存在系统性偏移。
- 某些配置字段、权限字段只落库不生效。
- 版主、搜索、配置历史等子模块的接口契约不完整或错误。
- 个别表已经建好，但没有对应 service/controller，形成“孤儿表”。
- 部分实现没有完全遵循仓库当前的 Drizzle 约束，后续维护成本会继续升高。

结论：论坛主链路已经不是“不可用”状态，但还不能认为“领域契约已收口”。建议优先处理 P0/P1，再做 DTO 和接口文档统一。

## 本次未复现的旧问题

- 未复现“主题删除误删非论坛评论”。
- 未复现“app 侧完全缺少 forum topic 独立接口”。
- 未复现“forum topic update 仍允许修改 `sectionId` / `userId`”。
- 未复现“论坛互动 resolver 仍按存在即合法，不校验公开可见性”。

## 关键问题

### P0. 版主创建/更新契约已经失真，`sectionIds` 被静默丢弃，超级版主权限还可能写入非法值

- 证据：
  - `libs/forum/src/moderator/moderator.service.ts:73`
  - `libs/forum/src/moderator/moderator.service.ts:77`
  - `libs/forum/src/moderator/moderator.service.ts:248`
  - `libs/forum/src/moderator/moderator.service.ts:261`
  - `apps/admin-api/src/modules/forum/moderator/moderator.controller.ts:22`
  - `apps/admin-api/src/modules/forum/moderator/moderator.controller.ts:61`
- 现状：
  - `CreateForumModeratorDto` / `UpdateForumModeratorDto` 都要求传 `sectionIds`，但 service 在创建和更新时都直接把它解构丢掉，没有落到 `forum_moderator_section`。
  - 超级版主权限使用 `Object.values(ForumModeratorPermissionEnum)` 组装；这里是数值枚举，运行时会同时带出字符串键和值，最终数组可能混入 `'PIN'`、`'FEATURE'` 这类非整数内容。
  - controller 文档全部声明返回 `ForumModeratorDto`，但真实返回分别是 `{ id }`、原始 moderator 行、`true`。
- 影响：
  - 调用方以为“创建版主时已绑定板块”，但数据库实际上没有绑定关系。
  - 超级版主权限数组可能写脏，进而影响权限判断或持久化。
  - Swagger 与真实行为完全不一致，前后端很难稳定协作。
- 维度：
  - 接口齐全性
  - 入参/回参合理性
  - 数据一致性
  - 代码质量
- 建议：
  - 创建/更新版主时要么同步写 `forum_moderator_section`，要么从 DTO 中移除 `sectionIds`。
  - 超级版主权限改成显式数字白名单，不要直接 `Object.values()` 数值枚举。
  - moderator controller 先把返回模型收敛到真实结构，再修 Swagger。

  - 回参与 DTO 一致性
  - 可维护性
- 建议：
  - 把 `BaseForumProfileDto` 收回到 `forum_profile` 表的真实字段。
  - 需要 app user / level 衍生信息时，新增组合 DTO，例如 `ForumProfileWithUserStateDto`。
  - admin topic detail 里的 `points/levelId/status` 改从 `topic.user` / `topic.user.level` 读取。

  - 直接写 `hits` 原对象，不要先 `JSON.stringify()`。
  - 如果历史数据已经落成字符串，建议补一次兼容迁移或读取兜底。

## 可维护性补充观察

- `ForumTopicService.incrementReplyCount()`、`incrementLikeCount()`、`decrementLikeCount()` 当前未被调用，容易误导维护者。
- `ForumSectionService.calculateStatistics()` 当前未被调用。

这些问题不会立刻打断业务，但会持续抬高后续排查成本。
