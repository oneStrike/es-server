# WebSocket 方案共识文档

## 目标定义

建设统一实时通信能力，支撑以下业务：
- 聊天室实时收发
- 用户发言被点赞通知
- 用户发言被评论通知

## 验收标准

### 功能验收

1. 用户连接后可基于 JWT 建立实时连接并绑定用户身份。
2. 聊天室支持加入、发言、离开、消息广播、ACK 回执。
3. 点赞与评论事件触发后，通知可落库并定向推送给目标用户。
4. 通知未读数可实时增量推送，客户端断线重连后可通过查询接口补齐。

### 质量验收

1. 关键链路具备幂等保护（消息发送、事件投递）。
2. 推送失败不导致通知丢失（落库为准，可重试可补偿）。
3. 基础安全能力覆盖：鉴权、限流、权限校验、载荷最小化。
4. 可观测能力覆盖：连接数、在线数、投递成功率、失败率、延迟分布。

## 技术实现共识

## 1) 架构形态

- 采用 NestJS WebSocket Gateway 作为实时入口。
- 采用“业务事务完成 -> 领域事件 -> 通知中心 -> WebSocket 投递”链路。
- 采用 Redis 承载用户连接索引与多实例消息分发能力。

## 2) 一致性策略

- 通知写库成功后再进行推送。
- 推送失败记录投递状态，异步重试。
- 客户端展示以查询接口返回为准，实时流只做增量加速。

## 3) 演进策略

- Phase A：单实例可运行闭环（本地/测试环境）。
- Phase B：启用 Redis Adapter 与 outbox 投递器（生产高可用形态）。

## 边界与限制

- 不引入外部消息队列作为前置依赖，优先使用现有 Redis。
- 不改变现有通知查询 API 的权威地位。
- 不在本阶段改造所有历史互动逻辑，仅接入点赞/评论主链路。

## 集成方案共识

- 鉴权复用现有 JWT 校验逻辑，握手阶段拒绝无效 token。
- 互动事件从现有点赞/评论服务发出，不在 Controller 层拼装。
- 通知中心统一编排标题、摘要、目标、来源、事件幂等键。
- WebSocket 层只负责连接、路由、投递，不承载复杂业务规则。

## 交付物共识

确认后按以下顺序编码：
1. 实时基础设施（网关、鉴权、连接管理、房间管理）
2. 聊天室闭环（协议、存储、广播、ACK）
3. 互动通知闭环（事件、通知落库、定向推送、未读增量）
4. 多实例增强（Redis Adapter、投递补偿、监控指标）
5. 测试与回归（单测/集成测试/接口联调）
