# WebSocket 方案对齐文档

## 原始需求

在当前项目中设计 WebSocket 使用方案，场景包括：
- 聊天室
- 通知用户发言被点赞
- 通知用户发言被评论

要求先输出可执行方案，确认后再编码。

## 项目现状分析

### 技术栈与运行形态

- 后端框架：NestJS + Fastify（app-api/admin-api 双应用）
- 数据层：Prisma + PostgreSQL
- 缓存：Redis
- 鉴权：JWT + Token 存储校验 + 黑名单机制

### 现有可复用能力

- 互动域已有点赞、评论、计数服务
- 已有论坛通知落库服务
- 已有进程内交互事件发射器（可扩展为实时通知事件源）
- 已有评论审核策略，可作为通知发送门禁

### 当前缺口

- 未落地 WebSocket 网关
- 未建立统一“通知投递总线”
- 未建立实时连接的用户路由与多实例分发机制

## 任务边界

### 本次包含

- 实时通信方案设计与执行蓝图
- 聊天室与互动通知的统一架构设计
- 鉴权、事件、协议、存储、可观测、灰度与回滚设计
- 任务拆解与验收标准定义

### 本次不包含

- 业务代码开发与数据库迁移执行
- 客户端 SDK 开发
- 大规模压测结果产出

## 核心约束

- 与现有 NestJS 模块化架构保持一致
- 复用现有 JWT 校验与 token 有效性判定链路
- 通知以“落库成功”为准，推送失败可补偿
- 聊天消息与通知必须支持幂等与重放安全
- 不能引入泄露敏感信息的事件载荷

## 关键假设

- 先在 `app-api` 内落地实时能力，不拆独立实时服务
- 客户端可在握手阶段携带 JWT
- 通知中心以用户维度推送，房间广播用于聊天室
- 评论待审核内容不立即触发对外通知

## 歧义消解与决策

1. 互动通知是否只覆盖论坛  
决策：V1 统一面向“发言”抽象，不限定论坛；论坛模块先落地，后续可扩展到内容评论。

2. 实时消息是否替代通知查询接口  
决策：不替代。实时推送是增量通道，通知列表查询仍为权威数据源。

3. 单实例还是多实例优先  
决策：方案按多实例兼容设计，实施分两阶段（先单实例闭环，再 Redis Adapter）。

## 验收入口定义

- 架构文档需明确：模块图、事件流、协议契约、数据模型、异常与补偿策略
- 任务文档需明确：每个任务输入输出、依赖、验收标准
- 清单文档需支持你逐项评审并给出确认意见
