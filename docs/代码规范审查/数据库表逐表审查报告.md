# 数据库表逐表审查报告

审查日期：2026-06-09（北京时间）

审查范围：`db/schema/**` 下 123 个 schema 文件、125 个 `snakeCase.table(...)` 表定义，关联阅读 `db/relations/**`、相关 migration SQL、以及高频服务查询路径。  
审查方法：人工逐域阅读表语义、字段、索引和服务访问路径；只用 `rg` / `Get-Content` 等定位证据，没有编写或运行自定义审查脚本替代判断。  
执行形态：按 work、forum/UGC、app 用户增长任务签到、支付会员优惠券广告表情配置、message/system/admin、全局 migration/分页/关系六个域并行审查后合并。  
限制说明：未连接生产/测试数据库执行 `EXPLAIN ANALYZE`，因此 `Using filesort/Using temporary` 类执行计划结论只作为结构性风险推断；必须用实际数据量和查询参数二次验证。

## 总体结论

**RECOMMENDATION: REQUEST CHANGES**

**Architectural Status: BLOCK**

阻断原因不是局部命名风格问题，而是数据库契约已被破坏：

- 仍存在物理外键，直接违反“绝对禁止在任何表中使用物理外键约束”。
- 多个被 `db/relations/**` 使用的关系字段缺少普通索引，违反“关系字段需建立普通索引”。
- 高频列表、日志、消息、评论、workflow、支付等路径大量默认 offset/count 分页，深分页和大表 count 风险未被接口契约隔离。
- 多张 append-heavy 表缺少明确分区、归档或冷热数据策略。
- 热表 migration 默认 SQL 使用普通 `ALTER/DROP INDEX/CREATE INDEX`，生产执行有锁表或 IO 峰值风险。

## 必须优先修复

### P0：物理外键违规

这些都应移除 schema 和 migration 中的物理 FK，只保留应用层事务、逻辑校验、补偿/对账和普通索引。

- `work_author_relation`：`db/schema/work/work-author-relation.ts:50`、`:55` 使用 `foreignKey(...)`；`db/migration/20260608143000_work_author_breaking_closure/migration.sql:56`、`:63` 创建 `FOREIGN KEY`。
- `emoji_asset`：`db/schema/app/emoji-asset.ts:183` 使用 `foreignKey(...)`；`db/migration/20260608013000_emoji_contract_hard_cutover/migration.sql:142` 创建 `FOREIGN KEY (pack_id)`。
- `emoji_recent_usage`：`db/schema/app/emoji-recent-usage.ts:77`、`:82` 使用 `foreignKey(...)`；`db/migration/20260608013000_emoji_contract_hard_cutover/migration.sql:150`、`:158` 创建 `FOREIGN KEY`。

建议增加守门规则：schema 禁止 import/call `foreignKey`，migration SQL 禁止出现 `FOREIGN KEY` / `REFERENCES`。

### P0：关系字段缺普通索引

### P1：分页和大表执行风险

- 全局分页 helper `db/core/query/page-query.ts:45` 统一计算 offset，缺少 cursor/keyset 契约和深分页上限。
- forum topic、comment、message notification、chat investigation、workflow、payment、audit 多数仍是 `count + limit + offset`。
- `libs/forum/src/search/search.service.ts` 的 ALL 搜索会扩大到 `offset + pageSize` 分别拉 topic/comment 后内存合并，深页风险更高。
- 建议：前台 feed、评论、消息、流水、日志、workflow 列表改 keyset/cursor；后台低频页保留 offset 时设置最大页深并延迟/条件化 count。

### P1：热表 migration 风险

- `db/migration/20260609103000_message_notification_chat_admin_repair/migration.sql:1`、`:16`、`:17`、`:35`、`:36`、`:47`、`:59` 使用普通 `ALTER TABLE`、`DROP INDEX`、`CREATE INDEX`。
- 同目录 README 已说明在线路径需要 `CREATE INDEX CONCURRENTLY` / `DROP INDEX CONCURRENTLY`，但默认 migration SQL 仍是阻塞式 DDL。
- forum/search 性能索引和 emoji cutover 也存在非并发建索引、批量 UPDATE、再加约束的上线风险。

## 冗余/低收益索引清单

优先删除或用 `EXPLAIN` 证明必须保留：

- 唯一约束 + 普通索引重复：`app_user.phoneNumber`、`app_user.emailAddress`、`app_user_token.jti`、`admin_user_token.jti`、`sensitive_word.word`、`work_category.name`、`work_tag.name`、`check_in_record(userId, signDate)`、`workflow_attempt(workflowJobId, attemptNo)`、`message_ws_metric.bucketAt`、`coupon_admin_grant_job.workflowJobId`。
- 前缀重复：`user_badge_assignment(userId, createdAt)` 被 `(userId, createdAt, badgeId)` 覆盖；`notification_delivery` 多组 status/receiver/category 前缀索引；`payment_order_provider_config_status_idx` 被更长 provider/status/createdAt 覆盖。
- 低选择性单列索引：性别、布尔启用、状态类字段在多张表中单列建索引，应改成实际过滤 + 排序组合或 partial index。

## 推荐修复顺序

1. 移除所有物理 FK，给 schema/migration 加守门规则，防止再次进入主干。
2. 补齐 relations 字段普通索引，优先 message/forum/payment/membership/coupon/task/admin 这些当前明确缺口。
3. 清理重复索引，降低写放大；每删一个索引用对应查询 `EXPLAIN` 验证。
4. 为高频前台列表和 append-heavy 表定义 cursor/keyset 分页契约，后台 offset 设置页深上限。
5. 为日志、消息、事件、流水表制定分区/归档/TTL 策略。
6. 将热表 migration 拆成在线 DDL 路径，默认生产迁移使用 concurrent index 流程或独立运维步骤。

## 最终门禁

在完成 P0 后仍不能直接通过，还需至少补齐 P1 中的分页/热表/冷热策略设计，否则数据量上来后会出现写入放大、深分页退化、日志表膨胀和上线锁表风险。
