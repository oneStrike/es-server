# App 前端破坏性更新清单

## 说明

- 本文只列 App 前端会直接感知的 interaction 模块合同变化。
- 本轮不做兼容层：旧查询参数、旧删除语义、旧返回形状立即失效。

## 1. 已购历史查询参数收口

- 影响接口：
  - `GET app/purchase/work/page`
  - `GET app/purchase/chapter/page`
- 变更内容：
  - 查询参数 `targetType` 删除，不再属于有效合同。
  - 继续保留 `status`、`workType`、`workId`、分页和日期范围字段。
- 前端必须调整：
  - 停止传递 `targetType`。
  - 不要再把 `targetType` 当成已购历史筛选条件。

## 2. 已购 / 已下载历史展示口径

- 影响接口：
  - `GET app/purchase/work/page`
  - `GET app/purchase/chapter/page`
  - `GET app/download/work/page`
  - `GET app/download/chapter/page`
- 变更内容：
  - 已购 / 已下载历史继续展示已被软删除的作品或章节记录。
  - 本轮不额外补 tombstone 字段，仍沿用当前响应结构。
- 前端必须调整：
  - 历史列表需要接受“作品或章节已下线，但历史记录仍展示”的结果。
  - 不要把“是否出现在历史里”再解释成“当前仍可访问”。

## 3. 评论删除语义

- 影响接口：
  - `POST app/comment/delete`
- 变更内容：
  - 删除根评论时，会级联删除该根评论下整棵回复树。
  - 删除单条回复时，仍只删除当前回复自身。
- 前端必须调整：
  - 不要再假设删除根评论后楼中楼回复还会继续保留。
  - 评论列表、回复数、删除后的本地状态要按“整棵树一起消失”处理。

## 4. 评论删除返回值

- 影响接口：
  - `POST app/comment/delete`
- 变更内容：
  - 返回值统一为 `Boolean`。
- 前端必须调整：
  - 不要再读取删除接口的 `id` 回显。
  - 删除成功只以布尔成功语义处理。

## 5. 不变项

- 购买、下载、评论相关接口的路径不变。
- 已购 / 已下载历史的响应主体结构不新增兼容字段。
- 评论创建、回复、分页查询的请求和响应结构本轮不做额外兼容改造。
