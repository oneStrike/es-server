# 对齐文档：评论模块统一重构

## 1. 原始需求
- 重构评论模块，统一支持：
  - 漫画评论
  - 小说评论
  - 漫画章节评论
  - 小说章节评论
  - 论坛帖子评论
- 覆盖范围从数据表到服务到控制器。
- 输出一份可确认的完整方案落地到本地文档。
- 有歧义点必须先征得确认。

## 2. 项目上下文分析

### 2.1 当前评论现状（双轨并存）
- 作品评论链路：`work_comment` + `WorkCommentService`，主要被 admin 内容管理使用。
- 交互统一评论链路：`user_comment` + `CommentService`，已具备多目标类型与校验器能力。
- 当前存在能力重复与口径不一致（审核默认值、计数字段、接口入口分散）。

### 2.2 目标类型与校验基础
- 已有统一目标类型枚举 `InteractionTargetType`，包含漫画/小说/章节/论坛主题。
- 已有 `TargetValidatorRegistry` 与各目标校验器，可复用做目标存在性与类型一致性校验。

### 2.3 论坛侧现状
- 论坛当前仍有 `forum_reply`、`forum_reply_like` 与相关服务链路。
- 论坛通知、举报、搜索、计数服务均与 `forum_reply` 有依赖，迁移需分阶段处理。

## 3. 已确认决策（来自本轮确认）
- 数据主表：统一以 `user_comment` 为唯一评论主表。
- 审核默认策略：由系统配置 `contentReviewPolicy` 决定，不写死固定默认值。
- 迁移策略：全量迁移 `work_comment` 历史数据并切主。
- 接口范围：同时覆盖 C 端与管理端。
- 计数策略：新增统一 `commentCount` 字段（包含 Work、ForumTopic）。
- 论坛回复关系：统一评论替代 `ForumReply`，迁移完成后下线旧链路。

## 4. 范围定义

### 4.1 本次包含
- 数据层：
  - `user_comment*` 成为唯一评论域主表族。
  - 新增并迁移必要字段与索引。
  - 迁移 `work_comment`、`forum_reply` 相关历史数据到统一评论域。
- 服务层：
  - 统一评论服务（创建、删除、查询、审核、隐藏、举报、点赞、计数）。
  - 统一计数策略与审计日志。
  - 与敏感词、系统配置、用户权限、行为日志集成。
- 控制器层：
  - C 端评论接口（发评、删评、列表、回复列表、点赞、举报）。
  - 管理端审核与治理接口（审核、隐藏、删除、举报处理、分页查询）。

### 4.2 本次不包含
- 评论内容推荐/排序算法升级（仅保留当前排序能力）。
- 跨服务拆分（仍保持在现有单体 Nest 模块边界内）。
- 前端页面实现细节（仅提供后端契约与迁移方案）。

## 5. 核心约束
- 严格对齐现有项目规范：Nest 模块化、DTO 校验、Prisma 模型规范、审计装饰器、统一异常模式。
- 不引入未在项目中验证的新框架。
- 数据迁移必须可回滚、可分批、可校验。
- 不中断线上写入，采用灰度切换策略。

## 6. 关键风险与应对
- 双写期一致性风险：通过事务内双写与对账任务保障。
- 历史数据口径差异（审核状态/楼层/回复关系）：在迁移脚本中显式映射规则并做抽样校验。
- 论坛侧依赖面广：分阶段替换服务入口，最后下线 `forum_reply`。
- 计数字段变更影响查询：先加字段与回填，再切换读路径。

## 7. 验收标准（对齐阶段）
- 目标类型全覆盖且无歧义。
- 数据主表、审核策略、迁移路径、接口范围、计数字段与论坛替代策略均已确认。
- 已形成可执行的共识、设计、任务拆分文档并可进入实现阶段。
