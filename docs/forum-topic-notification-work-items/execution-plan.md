# 论坛主题通知改造任务执行计划

## 1. 文档目标

本文用于把论坛主题通知改造拆分成“可排期、可并行、可交付”的执行计划。

注意：

- 本文是当前任务包里“优先级 / 依赖 / 波次 / 并行原则”的唯一事实源
- 其他总览文档与开发补充文档不再重复维护任务顺序

如需查看每个任务对应的预计改动模块、关键文件和测试点，请配合阅读：

- [development-plan.md](./development-plan.md)

## 2. 优先级定义

- `S0`：立即执行。直接影响通知类型语义、主链路正确性或后续任务是否返工
- `S1`：高优先。直接影响论坛主题通知的用户可见结果
- `S2`：中优先。属于本轮改造的重要补强，但不是当前 blocker
- `S3`：后置优先。价值明确，但可放在主链路稳定后推进

## 3. 依赖说明

- `硬前置`：不完成前通常不建议开工，否则会返工或语义不稳
- `软前置`：不是绝对 blocker，但建议先完成，能减少回改成本
- `直接后置`：该任务完成后最直接解锁的后续任务
- `可并行`：写入面和语义边界相对独立，可并行推进

## 5. 推荐实施波次

### Wave 1：先冻结通知类型与 outbox 契约

- `P0-01`

说明：

- 先把 `TOPIC_*` 类型和 `payload.type` 唯一事实源固定下来
- 这样后续不管是点赞、收藏、评论还是回复，都不会在半路换类型语义
- 本波只完成 `payload.type -> eventType` 的统一派生与 mismatch guard；`eventType` 字段的彻底删除另拆全仓清理任务

### Wave 2：收口通知构造层

- `P0-02`

说明：

- 先引入 composer 和展示快照契约
- 可以避免后续每个业务模块各自拼一次动态文案

### Wave 3：迁移已有互动通知

- `P1-01`
- `P1-02`

说明：

- 一个收口主题点赞 / 收藏
- 一个升级评论回复动态文案，并同步升级 `COMMENT_REPLY` 默认模板
- `P1-02` 在本波先补评论可见副作用共享载荷，供后续 `TOPIC_COMMENT` 复用
- 两者写入面相对独立，可并行推进

### Wave 4：补齐主题被评论通知

- `P1-03`

说明：

- 这一步会触及评论 hook 签名与首次可见补偿逻辑
- 建议在点赞 / 收藏 / 回复三条链路稳定后推进，并直接复用 `P1-02` 已补的评论副作用底座
- 同步完成 `TOPIC_COMMENT` 默认模板与联调 seed 升级，避免模板环境继续显示旧静态 copy

### Wave 5：收口主题模板层

- `P2-01`
- `P2-02`

说明：

- 先收口剩余 seed 示例与模板回退验收
- 再做缓存与保存期校验

## 6. 并行原则

建议只有在以下条件同时成立时再并行：

1. 没有硬前置依赖
2. 不会同时修改同一套通知类型定义
3. 不会同时改同一条评论补偿主链路
4. 不会同时改同一组模板配置与模板服务底座

不建议并行的典型组合：

- `P0-01` 与其他任一通知任务同时实施
- `P0-02` 与 `P1-03` 同时实施
- `P2-01` 与 `P2-02` 同时实施

## 7. 最终建议

这批任务最稳妥的推进方式是：

`先收口通知类型与构造层 -> 再迁移已有互动通知 -> 再补主题被评论 -> 最后补模板层可维护性`

只要顺序不反，论坛主题通知会越做越清晰；一旦顺序反了，就会回到“类型没收口、文案先乱改”的状态。
