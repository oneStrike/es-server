# Forum Topic Notification 方案生成规范

## 1. 目标

本规范用于统一 `forum-topic-notification-work-items` 目录下方案文档的生成方式，确保：

- 文档职责清晰，避免多处维护同一事实；
- 任务可拆解、可执行、可验收；
- 排期、依赖、验收证据具备可追踪性；
- 新增任务文档与现有风格保持一致。

## 2. 适用范围

- 适用于论坛主题通知相关的规划、开发、验收文档；
- 适用于新增 work item、调整优先级、补充验收标准、补录执行证据；
- 不用于替代业务设计文档或接口契约文档。

## 3. 文档分层与唯一事实源

### 3.1 文档职责分工

- `README.md`：目录说明、文档职责边界、阅读顺序；
- `execution-plan.md`：唯一排期事实源（优先级、依赖、波次、并行策略）；
- `development-plan.md`：开发执行补充（开工条件、改动模块、影响文件、测试点）；
- `p0/`、`p1/`、`p2/`：单任务实施单；
- `final-acceptance-checklist.md`：跨任务最终验收与上线阻塞项。

### 3.2 唯一事实源约束

- 排期顺序、波次、优先级仅允许在 `execution-plan.md` 维护；
- 任务文档可以引用排期，不得在任务文档内重新定义另一套排期；
- 若出现冲突，以 `execution-plan.md` 为准，并同步修正文档引用。

## 4. 命名规范

### 4.1 文件命名

- 任务目录按优先级分层：`p0/`、`p1/`、`p2/`；
- 任务文件命名：`NN-kebab-case.md`（`NN` 为两位序号）；
- 规范文档、总览文档使用语义化英文短横线命名。

### 4.2 标题命名

- 任务标题格式：`P{优先级}-{序号} 中文任务名`；
- 标题必须直接表达业务目标，不使用含糊词。

## 5. 方案生成流程

1. 先更新 `execution-plan.md`，确认优先级、依赖、波次；
2. 按优先级目录创建或更新任务文档；
3. 在 `development-plan.md` 补充开工条件、影响范围、测试重点；
4. 在 `final-acceptance-checklist.md` 新增或更新验收项与证据位；
5. 自检文档一致性（命名、引用、依赖关系、验收口径）。

## 6. 单任务文档固定模板

每个任务文档必须包含以下章节，且顺序保持一致：

1. `目标`
2. `范围`
3. `当前代码锚点`
4. `非目标`
5. `主要改动`
6. `完成标准`
7. `完成后同步文档`
8. `排期引用（对应 execution-plan 条目）`

## 7. 排期与依赖表达规范

- 使用 `S0 ~ S3` 表示优先级；
- 使用 `Wave N` 表示实施波次；
- 依赖关系需显式标注：`硬前置`、`软前置`、`直接后置`、`可并行`；
- 若任务可并行，需写明并行前提与互不阻塞条件；
- 不允许只写“待定”，必须给出可执行依赖描述。

## 8. 验收与证据规范

### 8.1 验收项写法

- 使用可勾选清单（checkbox）；
- 每条验收项必须可测试、可观察、可复核；
- 完成标准中至少覆盖：功能行为、边界条件、回归风险。

### 8.2 证据写法

- 每项验收需绑定证据类型：测试结果、日志、接口返回、截图或数据记录；
- 影响上线的风险必须落入“阻塞上线项”；
- 最终签收问题必须独立列出，不与普通备注混写。

## 9. 边界与禁忌

- 必须明确 `非目标`，防止范围蔓延；
- 不在任务文档中复制 `execution-plan.md` 的完整排期表；
- 不省略“完成后同步文档”，避免实施后文档失联；
- 不使用无法验证的表述，例如“基本完成”“大致可用”。

## 10. 质量门禁（发布前检查）

- 文档结构是否符合固定模板；
- 命名是否符合 `pN/NN-kebab-case.md`；
- 排期引用是否全部指向 `execution-plan.md`；
- 每个任务是否都有明确 `非目标` 与 `完成标准`；
- 验收清单是否包含证据位与阻塞上线项；
- README、排期、任务单、验收单之间是否一致。

## 11. 最小任务模板（可复制）

```md
# P1-01 示例任务名

## 目标

## 范围

## 当前代码锚点

## 非目标

## 主要改动

## 完成标准

## 完成后同步文档

## 排期引用
- 对应 execution-plan：Wave X / Sx / 依赖关系说明
```
