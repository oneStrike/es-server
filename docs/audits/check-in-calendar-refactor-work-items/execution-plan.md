# 签到模块改造执行排期

## 优先级说明

- `P0`：阻塞签到计划建模、奖励结算或 App / Admin 契约收口的任务，未完成前不进入整体验收。
- `P1`：依赖主链路稳定后再推进的验证与交付收口任务，不阻塞 `P0` 的主功能实现。

## 状态说明

- 状态仅反映当前分支的代码落地情况，不等同于最终签收结论。
- 状态取值约定：
  - `pending`：未开始
  - `in_progress`：已部分落地但仍有缺口
  - `completed`：代码已落地，待验收补证据
- 最终签收、验证证据与命令输出仍以 [final-acceptance-checklist.md](./checklists/final-acceptance-checklist.md) 为准。

## 依赖术语

- `硬前置`：必须完成后，当前任务才能开工。
- `软前置`：建议优先完成，但不满足时可在风险可控前提下并行推进。
- `可并行`：在无共享写集冲突时可并行执行。
- `直接后置`：当前任务完成后应立即衔接的任务。

## 任务依赖补充说明

- `P0-01` 负责唯一的数据模型、快照结构和按日奖励 owner，后续任务不得各自定义第二套日奖励结构或周期解释。
- `P0-02` 负责唯一的排期切换和计划边界校验口径，`P0-03` 只能复用其对外定义，不能在执行层再派生“即时切换”分支。
- `P0-02` 与 `P0-03` 都会修改 `libs/growth/src/check-in/check-in.service.support.ts`、DTO 和快照结构，默认串行推进，避免共享写集冲突。
- `P1-01` 必须在 `P0-02` 和 `P0-03` 都稳定后执行，避免验证证据建立在临时契约上。
