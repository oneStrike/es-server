# Task / Growth / Notification 全链路重估执行计划

## 1. 文档目标

本文是本轮 `task / growth / notification` 重构工作的唯一排期事实源。

本文负责回答：

1. 哪些任务必须先做
2. 它们之间如何依赖
3. 建议按什么波次落地
4. 当前状态与变更记录是什么

## 2. 优先级定义

- `S0`：不先做会持续放大模型歧义或直接影响后续所有实现方向
- `S1`：高优先，直接影响主链路可落地性与解释力
- `S2`：中优先，重要但可以在主合同稳定后继续推进
- `S3`：后置优先，适合在前面波次验证有效后再进入结构性升级

## 3. 依赖说明

- `硬前置`：未完成前不建议进入实施
- `软前置`：建议先完成，但不是绝对 blocker
- `直接后置`：最直接被解锁的后续任务
- `可并行`：语义独立、写入面冲突可控，可并行推进

## 3.1 当前默认决议

为避免工作包实施时再次分叉，本轮先固定以下默认决议：

1. producer 统一合同采用“`EventEnvelope + 稳定 bizKey`”双层结构：
   - `EventEnvelope` 继续沿用现有 `subjectId / operatorId / governanceStatus / context`；
   - `bizKey` 作为奖励幂等和任务推进去重的桥接字段，由统一事件桥接入口额外携带。
2. 事件任务的周期归属以 `eventEnvelope.occurredAt` 为准；仅在缺失时回退到消费时刻。
3. `EVENT_COUNT + MANUAL claimMode` 第一阶段默认不回补领取前事件，assignment 创建后才开始累计。
4. 任务通知第一阶段保持 `TASK_REMINDER` 主类型不拆分，`AVAILABLE / EXPIRING_SOON / REWARD_GRANTED` 通过 `payload.reminderKind` 收口。
5. 治理反转后的任务回滚第一阶段只保留审计与人工修复入口，不做自动回滚。

## 5. 推荐实施波次

### Wave 1：先定合同，再动主链路

- `P0-01`
- `P0-02`
- `P0-03`

说明：

- 这一波不追求先上事件驱动，而是先把 `task`、基础奖励、任务 bonus 的边界固定下来。
- 只要这三张任务单没完成，后续 `task`、`producer`、`notification` 改造都会反复返工。

### Wave 2：把通知和 producer 接线补平

- `P0-04`
- `P1-01`

说明：

- 先统一 task 提醒 payload、`reminderKind` 子语义、outbox 观测和通知结果语义。
- 再统一 producer 的奖励入口、事件壳与桥接派发入口，减少后续 Task consumer 的特殊分支。

### Wave 3：让任务真正吃到业务事件

- `P1-02`
- `P1-03`

说明：

- 在 producer 与目标模型稳定后，再进入事件驱动任务推进。
- 同波次把 App/Admin 读模型和运营视图一起收口，避免用户侧/后台侧继续暴露旧语义。

### Wave 4：进入结构性优化与运营治理

- `P2-01`
- `P2-02`
- `P2-03`

说明：

- 这波属于模型稳定后的增强项。
- 是否拆 `task_template / task_publish / campaign`，应基于前 3 波落地结果再决定，不提前透支复杂度。

## 6. 状态定义

- `pending`：已确认范围，尚未开始实施
- `in_progress`：已进入设计、代码或测试阶段
- `completed`：当前工作包范围内的代码、主文档与单测已收口；若涉及最终上线签收，仍以 `final-acceptance-checklist.md` 为准
- `blocked`：存在明确阻塞原因，暂不推进

## 8. 并行原则

1. `P0-02` 与 `P0-03` 可以在 `P0-01` 完成后并行，但最终落库字段和文案必须在同一轮 review 中收口。
2. `P1-01` 与 `P0-04` 可并行，但共享 `bizKey / payload / event context` 设计决策，不能各自扩展。
3. `P1-02` 开工前必须先确认“事件桥接入口”“事件发生时间落周期”和“`EVENT_COUNT + MANUAL` 是否回补”的合同已经在文档中固定。
4. `P2-02` 不应抢跑；只有在 `P1-02 / P1-03` 证明单表发布模型已成为真实瓶颈后才开工。

## 8.1 任务开工前复核要求

1. 除首个任务外，每个任务开工前必须先复核上一个已完成任务的完成标准、测试证据和文档同步是否准确。
2. 若发现上一个任务存在口径漂移、证据不足或实现偏差，必须先修正文档或代码，再开始下一个任务。
3. 复核结论应至少记录：
   - 是否满足上一任务完成标准
   - 是否存在遗留风险
   - 是否影响当前任务开工
