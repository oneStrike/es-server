# 项目计数器规范

适用范围：本仓库所有 `apps/*`、`libs/*`、`db/*`、`scripts/*` 中与业务计数、聚合读模型、冗余统计字段、浏览量、点赞量、收藏量、评论量、关注量、下载量、购买量、未读数相关的实现。

本文件只定义计数器的硬约束、决策顺序与验收基线。

- 实现模式与代码模板：`./COUNTER_PATTERNS.md`
- 当前仓库计数字段 owner / 事实表 / 修复状态：`./COUNTER_REGISTRY.md`
- Drizzle 技术约束：`./drizzle-guidelines.md`
- 项目级工作方式：`../../AGENTS.md`

若规范与当前可运行抽象、稳定接口契约、既有重试语义或基础设施入口冲突，以当前实现为准，并在交付说明中明确记录冲突点，而不是默默复制旧的不一致写法。

## 1. 文档体系与使用顺序

处理计数器相关改动时，按以下顺序决策：

1. 先在 `COUNTER_REGISTRY.md` 确认该字段属于哪一类计数、当前 owner 是谁、是否已有 repair。
2. 再在本文件确认硬约束是否满足。
3. 最后在 `COUNTER_PATTERNS.md` 选择实现模式或迁移方式。

如果注册表里没有该字段，先补齐字段归类和 owner，再开始写代码。

## 2. 计数器分类

### 2.1 事实计数

- 基于事实表运行时聚合得到的数量，不落库或只用于低频报表。
- 例：`count(*)`、`sum(...)`、用户资产概览中的实时统计。

### 2.2 对象冗余计数

- 冗余存放在目标表上的高频展示字段，用于详情页、列表页、排序和筛选。
- 例：`work.likeCount`、`forum_topic.replyCount`、`forum_section.followersCount`。

### 2.3 用户聚合计数

- 面向用户资料卡、用户中心、排行榜等场景的聚合读模型。
- 例：`app_user_count.followingCount`、`app_user_count.commentReceivedLikeCount`。

### 2.4 状态缓存计数

- 服务于当前用户会话状态的计数缓存，可在局部上下文中重算。
- 例：`chat_conversation_member.unreadCount`。

## 3. 必须满足的约束

### 3.1 事实表是真相

- 所有业务计数都必须能追溯到事实表或上游对象状态。
- 冗余计数字段默认是读模型，不得反向充当业务事实源。

### 3.2 同一字段只有一个 owner

- 同一计数字段只能有一个明确 owner 负责直接写入。
- 其他模块必须通过 owner service 或 owner 所暴露的统一写入口委托更新。
- 历史模块存在多个写入口时，新增代码不得继续复制这种模式。

### 3.3 分层职责固定

- `db/extensions` helper 只负责技术能力：原子增减、`returning` 校验、通用存在性约束、事务兼容。
- 领域 `CounterService` 负责计数字段口径、字段归属、联动关系、repair / rebuild。
- 业务 service 负责事实表写入和事务编排。
- Controller / Resolver 只负责入参、上下文和委托，不直接拥有计数字段写语义。

说明：

- 不是所有计数都必须新建独立 `CounterService`。
- 但只要某组计数字段已经形成“读模型 owner”，后续路径就必须继续复用该 owner。
- 状态缓存计数允许留在本域 service 中维护，但仍需满足事务、原子性和非负保护。

### 3.4 写路径必须事务内完成

- 事实表写入与相关计数更新必须在同一事务中完成。
- 事务必须沿调用链透传，禁止收到 `tx` 后继续偷偷使用默认 `db`。
- helper 可以运行在事务里，但不会代替上层自动开启事务。

### 3.5 增减必须原子更新

- 增减类计数必须使用数据库原子更新。
- 优先使用 `drizzle.ext.applyCountDelta` 或等价的 `update ... returning` 封装。
- 禁止“先查值、内存加减、再写回”的读改写流程。
- `delta === 0` 必须直接返回。

### 3.6 资源存在性与非负语义必须可判定

- 语义上要求目标存在的计数更新，必须校验受影响行数。
- 可减的计数字段必须防止下穿为负数。
- “目标不存在”和“计数不足”要能区分，不得静默吞掉。

### 3.7 口径必须显式

- 每个 count 字段都必须能回答：
  - 统计什么对象
  - 是否只统计可见数据
  - 是否包含软删、隐藏、待审核数据
  - 浏览类字段统计的是登录用户浏览、去重浏览，还是全量 PV
- `commentCount`、`replyCount`、`followersCount`、`followingCount` 等容易歧义的字段，必须在实现或文档中说明口径。

### 3.8 高频冗余计数必须可修复

- 高频对象冗余计数和用户聚合计数必须能基于事实表回填。
- 至少要有单条 rebuild 能力；高频核心计数应补充批量 repair。
- repair 不能基于另一份冗余计数链式修复。

### 3.9 新增入口不得扩散历史不一致

- 如果当前仓库存在散落 resolver 写 SQL、详情链路未接 browse log、repair 缺失等历史问题，新增代码必须朝统一 owner 收敛。
- 短期无法收敛时，要在交付说明里明确写出：
  - 沿用了哪段历史实现
  - 为什么这次不收
  - 后续建议的收敛方向

## 4. 推荐遵循的做法

### 4.1 Schema 与数据库兜底

- 持久化计数字段优先使用 `integer().default(0).notNull()`。
- 核心计数字段推荐补数据库级 `CHECK (count >= 0)`。
- 只有确实用于排序、筛选或榜单的 count 字段才建索引。

### 4.2 读模型更新可追踪

- 计数字段更新时，推荐同步刷新 `updatedAt`。
- 重要计数字段的 owner service 推荐提供结构化日志或 repair 日志。

### 4.3 优先使用领域 owner，而不是全局大而全 CounterService

- helper 应通用，owner service 应按领域划分。
- 不建议把用户、论坛、内容、消息全部塞进一个全局 `CounterService`。

## 5. 当前仓库的例外与边界

- 内容域当前仍存在若干 resolver 内部手写 delta 的历史路径，这属于待收敛现状，不是新增实现可以继续复制的模式。
- 浏览计数当前并不天然等于全量 PV；现有 browse log 语义是“行为事实 + 当前接入链路能覆盖到的浏览”。
- `chat_conversation_member.unreadCount` 这类状态缓存计数可以由本域 service 维护，不强行要求抽成通用 CounterService。
- 系统监控、重试、命中次数等运维型计数不作为本文件的首要治理对象，但仍应遵守原子更新与事务一致性的技术原则。

## 6. 验收清单

- [ ] 已确认该字段属于事实计数、对象冗余计数、用户聚合计数或状态缓存计数。
- [ ] 已确认事实来源、统计口径和当前 owner。
- [ ] 同一计数字段没有新增第二个平级写入口。
- [ ] 事实表写入与计数更新在同一事务中完成。
- [ ] 增减使用原子更新，没有读改写。
- [ ] 存在性和非负语义有明确校验。
- [ ] DTO、注释或文档体现了关键口径差异。
- [ ] 高频冗余计数具备基于事实表的 repair 方案，或在交付中明确记录缺口。
- [ ] 若沿用了历史不一致，说明中已写明冲突点与收敛建议。
