# 项目计数器规范

适用范围：本仓库所有业务计数、聚合读模型、冗余统计字段与状态缓存计数。

## 1. 文档定位

- 本文件只定义计数器硬约束与验收基线。
- 实现模式参考 `../references/COUNTER_PATTERNS.md`。
- 当前字段 owner、事实来源与修复状态参考 `../references/COUNTER_REGISTRY.md`。

## 2. 基本分类

- 事实计数：基于事实表实时聚合，不把冗余字段当事实源。
- 对象冗余计数：存放在目标对象上的高频展示字段。
- 用户聚合计数：面向用户中心、资料卡、排行榜等场景的聚合读模型。
- 状态缓存计数：服务于当前用户上下文、允许按局部上下文重算的计数。

## 3. 硬约束

- 所有业务计数都必须能追溯到事实表或上游对象状态。
- 同一计数字段只能有一个 owner；其他模块必须通过 owner 暴露的统一入口更新。
- `db/extensions` 只负责通用技术能力；计数字段口径、归属、联动与 repair 由领域 owner 负责。
- 事实写入与计数更新必须在同一事务中完成，事务必须沿调用链透传。
- 增减类计数必须使用原子更新，禁止“先查再算再写回”。
- `delta === 0` 直接返回。
- 语义上要求目标存在的计数更新必须校验受影响行数。
- 可减计数字段必须防止下穿为负数，并能区分“目标不存在”和“计数不足”。
- 每个计数字段都必须有明确口径：统计对象、可见性、是否包含软删/隐藏/待审核，以及浏览类字段的去重或 PV 语义。
- 高频对象冗余计数和用户聚合计数必须具备基于事实表的 rebuild / repair 能力。
- 新增入口不得复制历史散落写法；短期无法收敛时，在交付说明中明确冲突点与后续收敛方向。

## 4. 推荐做法

- 持久化计数字段优先使用 `integer().default(0).notNull()`。
- 核心可减字段优先补数据库级非负保护。
- 只有排序、筛选、榜单真正需要的计数字段才建索引。
- 重要计数字段更新推荐输出结构化日志。

## 5. 验收清单

- [ ] 已确认该字段属于事实计数、对象冗余计数、用户聚合计数或状态缓存计数。
- [ ] 已确认事实来源、统计口径与唯一 owner。
- [ ] 同一计数字段没有新增第二个平级写入口。
- [ ] 事实写入与计数更新在同一事务中完成，且增减使用原子更新。
- [ ] 存在性与非负语义有明确校验。
- [ ] 高频冗余计数字段具备基于事实表的 repair / rebuild 方案，或在交付中明确缺口。
