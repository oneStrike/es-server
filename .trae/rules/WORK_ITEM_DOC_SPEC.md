# 项目方案与清单文档规范

适用范围：本仓库 `docs/*` 下所有用于需求落地的方案、排期、任务单、开发补充、验收清单文档。

## 1. 核心原则

- 文档分层清晰：每类文档只承载一种主要职责，避免“一文多责”。
- 唯一事实源：排期、依赖、状态等关键信息只能在指定文档维护。
- 可执行性优先：每个任务都必须具备明确输入、输出、完成标准与验收证据。
- 可追溯性优先：任务与排期、开发实现、验收结果之间必须可双向追踪。
- 小步演进：允许增量补充，但不得破坏既有文档契约和阅读顺序。

## 2. 标准文档集

每个工作包目录建议至少包含：

- `README.md`：文档集说明、阅读顺序、职责边界；
- `execution-plan.md`：排期唯一事实源（优先级、依赖、波次、并行策略）；
- `development-plan.md`：开发执行补充（开工条件、影响模块、测试重点）；
- `p0/`、`p1/`、`p2/`：分优先级任务单目录；
- `final-acceptance-checklist.md`：跨任务最终验收、阻塞上线项与签收结论。

若任务规模较小，可裁剪目录层级，但仍需保留“排期事实源 + 任务单 + 验收清单”三类核心文档。

## 3. 唯一事实源规则

- 任务优先级、执行波次、依赖关系仅在 `execution-plan.md` 维护。
- 单任务文档允许引用排期，不得复制或维护第二套排序。
- 开发计划只补充执行信息，不改写排期结论。
- 验收清单只记录验收结论与证据，不反向定义新范围。
- 文档冲突时，优先修复事实源文档，再统一更新引用文档。

## 4. 命名规范

### 4.1 目录命名

- 工作包目录使用业务语义的 `kebab-case`，例如 `forum-topic-notification-work-items`。
- 任务目录使用 `p0`、`p1`、`p2` 等优先级层级。

### 4.2 文件命名

- 任务文件统一：`NN-kebab-case.md`（`NN` 为两位顺序号）。
- 计划、验收、说明文件使用稳定语义名，不使用临时后缀如 `new`、`final2`。

### 4.3 标题命名

- 单任务标题推荐：`P{优先级}-{序号} 中文任务名`。
- 标题必须表达业务动作与目标对象，禁止含糊表述。

## 5. 文档模板要求

### 5.1 单任务文档固定章节

每个任务文档必须按以下顺序编排：

1. `目标`
2. `范围`
3. `当前代码锚点`
4. `非目标`
5. `主要改动`
6. `完成标准`
7. `完成后同步文档`
8. `排期引用`

### 5.2 排期文档最小结构

- 优先级说明（如 `S0 ~ S3`）；
- 波次划分（如 `Wave 1 ~ N`）；
- 依赖定义（硬前置、软前置、可并行、直接后置）；
- 每个任务的状态字段与变更记录入口。

### 5.3 验收清单最小结构

- 分阶段验收项（功能、回归、性能或稳定性）；
- 证据记录位（测试输出、日志、截图、接口响应、脚本结果）；
- 阻塞上线项；
- 最终签收问题与结论。

## 6. 依赖与状态表达规范

- 优先级、波次、依赖术语必须在同一文档集中保持一致。
- 依赖关系必须可执行，不得出现不可验证的“待定依赖”。
- 可并行任务必须注明并行前提、资源冲突约束和汇合节点。
- 状态字段建议至少包含：`pending`、`in_progress`、`completed`、`blocked`。
- 被阻塞任务必须附阻塞原因、影响范围与解除条件。

## 7. 验收与证据规范

- 每条验收项必须满足“可测试、可观察、可复核”。
- 验收标准优先写行为结果，不写主观描述。
- 涉及接口契约变更时，必须提供入参、出参、错误语义的对照证据。
- 涉及数据改动时，必须提供样例数据或脚本执行结果证据。
- 线上发布前必须显式确认“无阻塞上线项”或列出豁免说明。

## 8. 边界与禁忌

- 禁止在多个文档重复维护同一排期信息。
- 禁止省略 `非目标`，导致范围失控。
- 禁止把任务拆分成无法独立验收的“动作碎片”。
- 禁止使用“基本完成”“大概可用”等不可验收语言。
- 禁止只记录结论不留证据。

## 9. 维护与变更流程

1. 先更新 `execution-plan.md` 的任务排序与依赖；
2. 再更新对应任务单与开发补充；
3. 实施完成后更新 `final-acceptance-checklist.md`；
4. 合并前执行文档一致性检查；
5. 变更涉及规则冲突时，在交付说明中标记冲突点与暂行决策。

## 10. 验收清单

- [ ] 文档集具备 `README / execution-plan / tasks / acceptance` 的最小闭环。
- [ ] 排期、依赖、状态只有唯一事实源。
- [ ] 每个任务文档包含固定章节，且包含明确 `非目标`。
- [ ] 验收项具备可复核证据，不存在主观性结论。
- [ ] 阻塞上线项与最终签收结论已独立记录。
- [ ] 文档命名与目录命名符合规范，无临时命名污染。
