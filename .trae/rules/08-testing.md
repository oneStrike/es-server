# 测试规范

适用范围：`libs/*` 与 `apps/*` 中的单元测试、模块级测试与集成验证。

## 仓库约定

- 测试优先验证对外行为、错误语义、状态变化和数据一致性，不围绕私有实现细节写脆弱断言。
- `*.spec.ts` 是长期资产；成熟测试文件默认保留，不因任务完成而删除。
- 临时探针、一次性验证脚本、临时调试代码必须删除；不要把它们伪装成正式测试留在仓库里。
- 本仓库当前可用的默认测试面是 `pnpm test` 与按路径运行的 Jest；`pnpm test:e2e` 仍指向缺失配置，不作为默认验证命令。

## 默认动作

- 改动涉及行为、接口 contract、错误语义、事务语义、分页、时间逻辑、幂等、快照映射、计数器同步时，默认补测试或扩测试。
- 纯文档、纯注释、纯格式化变更通常不新增测试，但仍要跑必要的基线检查。
- cleanup / refactor / deslop 任务，默认先确认是否已有行为保护；没有的话，先补回归测试，再改实现。
- 优先写离改动最近的 `*.spec.ts`；不要把所有逻辑都推到一个巨型测试文件里。

## 何时必须补测试

- 新增或修改对外可见行为。
- 调整错误码、异常类型、错误 message、降级策略。
- 涉及事务、并发、重试、补偿、幂等、原子更新、计数同步。
- 调整分页、排序、筛选、默认值、时间窗口或时区语义。
- DTO / mapper / serializer / projection / notification payload 变更。
- 修复回归 bug，且缺少可复现保护时。

## 测试层次

- 纯函数、mapper、工具方法、值域转换、格式化逻辑：优先单元测试。
- service / resolver 的业务规则、错误语义、事务行为、数据库交互结果：优先模块级或集成式 `spec`。
- 只有在当前测试层无法有效覆盖时，才考虑更重的端到端验证。
- 当前仓库不把 `test:e2e` 当默认路径；不要把“没法跑 e2e”当作不写测试的理由。

## 如何写

- 测试名应直接描述行为结果，而不是实现步骤。
- 优先显式断言关键字段、错误码、状态变化和副作用；不要滥用大而脆的快照。
- 与时间相关的逻辑要固定时间源或使用稳定输入，避免测试漂移。
- 与随机值、排序、分页相关的断言要写清预期顺序和边界。
- 对错误路径至少覆盖一种代表性失败场景，明确断言错误语义。
- 若一个模块已有现成 spec，优先在原 spec 扩展；不要平铺重复测试文件。

## 默认验证命令

- 首选：`pnpm test -- --runInBand --runTestsByPath <spec-path-1> <spec-path-2>`
- 基线：`pnpm type-check`
- 共享逻辑、跨模块联动或影响面较大时：`pnpm test`
- `pnpm test:e2e` 不作为默认验证命令。

## 禁止项

- 禁止在行为变更后完全不补测试，也不运行现有相关测试。
- 禁止删除成熟 `*.spec.ts` 只为让当前任务“更干净”。
- 禁止把一次性脚本、手工验证代码或临时日志提交为正式测试。
- 禁止只断言实现细节，例如私有方法调用次数、局部变量中间值、无业务意义的内部步骤。
- 禁止在未运行验证命令的情况下声称“测试通过”“已修复”“已完成”。

## 正反例

- 允许：为 mapper 增加 `*.spec.ts`，断言公开返回结构与异常分支。
- 允许：改动 service 的事务逻辑后，补一个覆盖计数同步和资源不存在分支的 spec。
- 允许：只改一个 mapper 时，运行 `pnpm test -- --runInBand --runTestsByPath <that-spec>` 加 `pnpm type-check`。
- 禁止：改了分页默认排序，却只跑 `pnpm type-check` 不跑相关 spec。
- 禁止：为了省事删除现有失败 spec，而不是修正实现或测试数据。
- 禁止：把整个大对象直接快照，导致一点无关字段变化就让测试失真。
