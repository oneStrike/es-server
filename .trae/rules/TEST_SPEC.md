# 测试规范

适用范围：本仓库 `apps/*`、`libs/*`、`db/*` 下所有新增或修改行为，以及为这些行为补充的 Jest 测试。

## 1. 目标与原则

- 行为优先：优先验证对外行为、错误语义、状态流转与数据契约，不为凑覆盖率测试临时实现细节。
- 分层验证：根据改动所在层选择最小但有效的测试层级，避免所有问题都堆到 E2E。
- 仓库对齐：测试目录、命名、Mock 方式、验证命令应与本规范保持一致；若仓库中存在旧位置或旧命名，触碰相关模块时优先顺手收敛到当前规范。
- 风险优先：事务、分页、时间语义、幂等、快照冻结、奖励结算、计数器同步等高风险逻辑必须有测试。
- 证据优先：声称“已修复”“已兼容”“已防回归”时，必须给出对应测试或可复核的验证输出。

## 2. 测试分层

### 2.1 单元 / 契约测试

适用于：

- 纯函数、时间工具、格式化工具、枚举兼容逻辑；
- DTO 校验、默认值、移除枚举值兼容；
- Controller 元数据、路由路径、审计装饰器；
- Service 中稳定且高价值的业务 helper 契约。

要求：

- 默认优先使用这一层；
- 测试应快、稳定、可在 `--runInBand --runTestsByPath` 下独立执行；
- 不依赖真实数据库、真实网络和真实定时器。

### 2.2 领域行为测试

适用于：

- `libs/*` 下 Definition / Runtime / Execution 类服务；
- 状态机、奖励结算、快照冻结、分页映射、补偿逻辑；
- 需要验证事务内外职责边界但又不必拉起完整应用的场景。

要求：

- 通过最小化 Drizzle / 依赖 Mock 组合验证公开方法；
- 当私有或受保护方法本身承载稳定业务契约，且通过公开入口难以精确覆盖时，允许像现有仓库一样使用 `(service as any)` 做合同级测试；
- 只允许测试“稳定业务规则 helper”，不测试纯转发或无语义的临时拆分函数。

### 2.3 集成测试

适用于：

- Drizzle 查询拼装、唯一约束、检查约束、事务回滚、乐观锁；
- 依赖 `findPagination`、`withTransaction`、`withErrorHandling` 的关键行为；
- 单元测试无法证明正确性的 SQL / ORM 行为。

要求：

- 只覆盖最关键路径，不追求把所有分支都搬到集成层；
- 优先验证“单元测试最容易漏掉”的问题，例如唯一约束、分页排序、事务边界、持久化状态一致性。

### 2.4 E2E 测试

适用于：

- 登录鉴权、全链路路由、跨模块集成、全局拦截器或过滤器联动；
- 明确需要验证 HTTP 层行为的改动。

当前约束：

- `pnpm test:e2e` 当前仍指向缺失配置，修复前不作为默认验证命令；
- 新需求不要默认要求补 E2E，除非单元 / 集成测试无法覆盖风险。

## 3. 目录与命名约定

### 3.1 文件放置

- `libs` 领域服务测试：放在相邻 `test/` 目录，例如 `libs/growth/src/task/test/task.service.spec.ts`。
- `apps` 模块测试：默认放在模块内专门的 `test/` 目录，覆盖 Controller、Service、Guard、模块契约等，例如 `apps/app-api/src/modules/auth/test/auth.controller.spec.ts`、`apps/app-api/src/modules/auth/test/auth.service.spec.ts`。
- `apps` DTO 契约测试：优先放在模块 `test/` 目录，文件名明确体现 DTO 契约主题，例如 `apps/admin-api/src/modules/check-in/test/check-in.dto-contract.spec.ts`；仅当模块已有稳定 `dto/test/` 结构且继续沿用更清晰时，才保留在 `dto/test/`。
- 平台工具测试：放在相邻 `test/` 目录，例如 `libs/platform/src/utils/test/time.spec.ts`。
- 数据库 schema 契约测试：优先放在对应领域测试文件中，围绕表约束名和状态一致性做断言。
- 历史遗留的同级 `*.spec.ts` 可以暂时保留，但新增测试与被迁移测试默认收口到 `test/` 目录，不再继续扩散同级放置模式。

### 3.2 命名规则

- 测试文件统一使用 `*.spec.ts`。
- `describe` 使用“模块 + 主题”命名，例如 `check-in execution response contract`。
- `it` 使用完整行为句，例如 `preserves idempotent flags when the sign record already exists`。
- 同一文件内按“公共边界 -> 业务契约 -> 异常路径 -> schema 约束”的顺序组织。

## 4. 编写规则

### 4.1 Mock 规则

- 只 Mock 你不拥有的边界，例如 `@db/core`、消息模块、成长账本、第三方网关。
- 不要为了断言调用次数而深度 Mock 本地实现细节，除非调用次数本身就是业务契约。
- 涉及 `jest.mock(...)` 的服务测试，优先在 Mock 完依赖后再 `await import(...)` 目标模块，保持与仓库现有写法一致。
- 对复杂事务链，优先封装最小测试桩函数，例如 `createUpdateTransactionHarness`、`createCheckInDrizzleMock`，不要在每个 `it` 里重复拼链式 Mock。

### 4.2 断言规则

- 每个测试聚焦一个概念，不在一个 `it` 中混合多个无关断言。
- 优先断言用户可见或调用方可见结果：返回值、异常信息、状态字段、排序结果、写表载荷、审计元数据。
- 对时间、自然日、周期切片逻辑，必须显式固定 `process.env.TZ = 'Asia/Shanghai'` 或使用仓库时区工具，禁止依赖机器默认时区。
- 涉及兼容窗口或移除旧值时，必须同时覆盖“接受新值”和“拒绝旧值”。

### 4.3 允许直接测私有 / 受保护 helper 的场景

仅在以下条件同时满足时允许：

- helper 承载稳定业务契约，例如周期切片、快照冻结、版本升级判断、奖励候选解析；
- 通过公开方法覆盖该 helper 会引入大量无关 Mock，导致测试信号变差；
- 断言的是业务结果，而不是实现步骤。

不允许的场景：

- 纯转发方法；
- 无业务语义的映射胶水；
- 只因为“这样更容易写测试”而去测试临时拆分函数。

## 5. 各层必测清单

### 5.1 Controller

薄 Controller 至少覆盖以下内容：

- 路由路径或路径片段；
- 关键装饰器元数据，例如审计、权限、分页模型；
- 对 `libs/*` DTO / Service 的公开契约消费是否正确。

如果 Controller 出现业务编排，先考虑下沉到 `libs/*`，再为下沉后的服务补行为测试。

### 5.2 DTO

至少覆盖以下内容：

- `class-validator` 接受合法输入；
- 拒绝非法枚举、非法 JSON 结构、非法数值边界；
- 删除旧枚举值或旧字段兼容时，补回归测试；
- Query DTO 与对应服务查询入参保持 1:1。

### 5.3 Service

至少覆盖以下内容：

- 成功主链路；
- 至少一个核心异常分支；
- 幂等、重复提交、重复奖励、唯一键冲突；
- 时间窗口、发布窗口、状态流转、补偿重试；
- 默认排序、稳定排序、分页条件、摘要映射；
- 涉及快照的逻辑必须覆盖“使用快照而不是最新配置”。

### 5.4 Schema / 数据约束

当变更 `db/schema/*` 时，至少覆盖以下内容：

- 新增或修改的 `check(...)` 约束名称；
- 关键 `unique(...)` 约束名称；
- 状态字段一致性约束；
- 正向状态与非法状态的边界语义。

### 5.5 集成行为

以下任一场景，至少补 1 个集成测试：

- 事务内写多张表；
- 依赖数据库约束保证正确性；
- 并发或乐观锁丢失分支；
- 单元测试无法证明的分页 / 过滤 / SQL 行为；
- 账本、通知、Outbox、补偿等跨模块联动。

## 6. 覆盖率策略

- 不以全局 100% 为目标，也不允许“关键流程几乎没测”。
- 新增模块必须随 PR 一起补基础测试，不接受“先上功能，后补测试”的长期欠账。
- 对修改文件，推荐达到以下目标：
  - 纯工具或纯规则模块：语句 / 分支覆盖率尽量达到 90% 以上。
  - 领域服务模块：触达所有公开方法、所有关键状态流转、所有核心异常分支；触达率建议不低于 80%。
  - Controller / DTO：重点看关键契约是否覆盖，不强求高语句覆盖率。
- 若因基础设施限制无法提高覆盖率，需在交付说明中明确剩余风险，而不是只报一个百分比。

## 7. 默认验证命令

### 7.1 目标测试

```bash
pnpm test -- --runInBand --runTestsByPath <spec-path-1> <spec-path-2>
```

适用场景：

- 单模块修改；
- 本地快速回归；
- 修复单一失败用例。

### 7.2 定向覆盖率

```bash
pnpm test -- --runInBand --coverage --runTestsByPath <spec-paths...> --collectCoverageFrom="<file-path>"
```

适用场景：

- 判断某个高风险模块是否真的测到；
- 补测试时确认盲区。

### 7.3 仓库基线验证

```bash
pnpm type-check
pnpm test
```

适用场景：

- 跨模块改动；
- 调整公共 DTO、公共工具、共享 service support；
- 合并前最终回归。

### 7.4 非默认命令

```bash
pnpm test:e2e
```

说明：

- 当前不作为默认要求；
- 只有修复了缺失配置，并且本次改动确实需要 HTTP 全链路验证时才纳入。

## 8. 反模式

- 用 Snapshot 代替行为断言，导致失败时无法判断真实语义。
- 只测 happy path，不测异常、补偿、幂等和边界值。
- 把所有测试都堆到 Controller 层，跳过 `libs/*` 的核心业务验证。
- 断言过多内部调用细节，导致正常重构也频繁改测试。
- 忽略时区，直接拿本机自然日做断言。
- 为了凑覆盖率测试纯样板代码，反而不补高风险链路。

## 9. 提交前检查清单

- [ ] 改动对应的公开行为已有测试覆盖。
- [ ] 关键异常分支和边界值已有测试覆盖。
- [ ] 事务、幂等、分页、快照、时间语义等高风险逻辑已补测试。
- [ ] 目标测试命令已运行并通过。
- [ ] 涉及公共契约或跨模块改动时，已运行 `pnpm type-check` 与必要范围的 `pnpm test`。
- [ ] 如存在未覆盖风险，已在交付说明中明确记录。
