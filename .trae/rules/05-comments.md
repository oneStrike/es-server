# 注释规范

适用范围：`libs/*`、`apps/*` 与 `db/schema` 中的代码注释。

## TL;DR

- 何时看：改方法注释、导出类型 / 常量 / 枚举注释、schema 字段说明时先看本篇。
- 必做：每个方法定义前都写简短注释；导出稳定类型、常量字段、枚举成员和闭集 `smallint` 字段写清业务语义。
- 不要：用注释复述代码、堆模板化空注释、只写整体注释不写字段 / 成员语义，也不要用注释掩盖结构问题。
- 最低验证：代码改动跑对应基线检查；纯文档注释改动至少跑对应 Markdown 检查。

## 默认动作

- 本仓库不追求“注释覆盖率”；注释的目标是帮助维护者快速判断原因、约束、边界、语义和风险。
- 注释不逐句翻译代码。
- 每个方法定义前都必须有一条简短注释，直接说明“这个方法是做什么的”。
- 除方法注释外，代码本身已经足够清晰时，默认不补额外注释。
- 不要为了“看起来完整”写模板化空注释。
- 复杂 Service / Helper 若方法过长、职责纠缠，先整理结构、拆分命名，再补注释。
- 不要用大段注释掩盖结构问题。
- 默认使用简体中文，专有名词、框架名、协议名保留英文。

## 注释形式

- 导出稳定符号、schema 字段说明，优先使用 JSDoc。
- 导出的公共方法/函数可使用 JSDoc；私有方法、类内 helper 方法统一使用紧邻方法定义的行注释，不使用 JSDoc。
- 局部实现原因、特殊分支、性能取舍、兼容性 workaround，优先使用紧邻代码的行注释。
- 同一符号只保留一组有效注释；禁止为同一个类型、方法、字段堆叠多段重复 JSDoc。
- DTO 字段语义优先通过装饰器 `description`、`example`、校验参数表达；这些信息已经完整时，通常不需要额外字段注释。
- DTO 字段装饰器里的短文案默认内联；不要为了局部复用把它们再抽成 `*_DESCRIPTION`、`*_EXAMPLE` 一类常量。

## 必须写注释的场景

- 所有方法定义，包括公共方法、私有方法、类内 helper 方法。
- `*.type.ts` 中导出的稳定领域类型，包括 owner 模块 `types/` 目录中的 `*.type.ts`。
- 导出的常量对象、映射常量、配置常量中的每一个字段，以及导出枚举的每一个成员。
- 事务、幂等、重试、补偿、事件投递、原生 SQL、跨模块兼容逻辑等关键语义。
- `db/schema/**/*.ts` 中所有闭集 `smallint` / `smallint[]` 字段。
- `priority`、`sortOrder`、`dailyLimit`、`postInterval` 这类使用 `smallint` 的非枚举配置字段。
- 安全限制、权限前提、缓存一致性、性能优化等“代码看不出原因、后续修改容易踩坑”的地方。

## 如何写

- 方法注释必须写在方法定义正上方，优先用 1 行到 2 行中文行注释说明“它做什么”；确有必要时再补关键前提、副作用或限制。
- 类型注释优先写：它代表什么业务语义、供谁复用、和 DTO / schema 的关系是什么。
- 常量对象注释优先写：常量整体用途，以及对象内每一个字段各自表示什么。
- 枚举注释优先写：枚举整体用途，以及每一个成员各自对应的业务语义；闭集数值枚举要能直接看出值域含义。
- schema 字段注释优先写：字段业务含义、数值语义、默认值语义、边界约束。
- 常量字段注释必须紧邻字段本身；只给常量整体写注释、不给字段写注释，不算完成。
- 枚举成员注释必须紧邻成员本身；只给枚举整体写注释、不给成员写注释，不算完成。
- 若注释只是在复述函数名、字段名、`if` 条件或 SQL 字段映射，默认删掉。
- 若注释与实现不一致，应立即修正或删除；不要保留历史描述。

## 数据表字段注释要求

- 闭集状态 / 类型 / 模式 / 角色 / 平台 / 目标 / 场景字段，JSDoc 必须写清数值语义。
- `smallint` / `smallint[]` 字段统一写成 `1=...`、`2=...` 这类中文说明，不要求读者跳转业务代码反推。
- 对 `priority`、`sortOrder`、`dailyLimit`、`postInterval` 等非枚举 `smallint` 字段，必须写清 `0` 或默认值的业务含义，例如 `0=默认优先级`、`0=不限制`、`0=默认排序`。
- 若字段是开放字符串、事件键、模板键、路由键、MIME 类型、来源快照或其他非闭集业务键，应继续保持字符串，并在注释中说明这是开放值。
- 不要把开放业务键硬写成伪枚举说明，也不要把闭集数值字段只写成“状态”“类型”“模式”这类空泛描述。

## 禁止项

- 禁止模板化空注释、逐行翻译、与实际行为不一致的历史描述。
- 禁止对显而易见的 getter、setter、参数透传、简单字段赋值补“说明性废话”。
- 禁止为私有方法、类内 helper 方法使用 JSDoc；禁止用 `/** 获取用户 */` 这类复述代码的 JSDoc 作为方法注释。
- 禁止只给常量对象写整体注释，却不给对象内字段逐项写注释。
- 禁止只给导出枚举写整体注释，却不给成员逐项写注释。
- 禁止为同一符号堆叠重复注释，例如一段“业务解释”后再跟一段“稳定领域类型”模板。
- 禁止在 `EnumProperty.description`、schema 字段注释或业务字段注释中直接写英文常量名、旧字符串枚举值或技术 key。
- 禁止把 DTO 装饰器短文案机械提取成局部 `*_DESCRIPTION`、`*_EXAMPLE` 常量，再为这些常量补无意义注释。
- 禁止用注释替代命名、结构拆分和类型收敛。

## 正反例

- 允许：在方法定义前写 `// 批量获取主题列表使用的发帖用户简要信息。`
- 允许：为事务方法说明“写操作与计数更新必须位于同一事务中，否则会出现状态和计数不一致”。
- 允许：为稳定领域类型说明“仅供内部服务链路复用，避免重复定义”。
- 允许：为导出常量对象的每个字段逐项补注释，而不是只写对象整体用途。
- 允许：为 schema 字段写 `阅读规则（-1=继承作品，0=所有人可见，1=登录用户可见，2=会员可见，3=需购买可见）`。
- 允许：为开放字符串键写“开放值，由事件定义表约束，不在 schema 层枚举收敛”。
- 禁止：用 `/** 获取用户 */` 这种 JSDoc 作为方法注释。
- 禁止：只在 `AuthErrorMessages` 上方写“认证错误提示”，但不解释 `TOKEN_INVALID`、`FORBIDDEN` 等字段含义。
- 禁止：`/** 获取用户 */`、`/** 设置状态 */`、`// 判断是否为空` 这类复述代码的注释。
- 禁止：同一个 `interface` 或 `type` 连续写两段表达相近的 JSDoc。
- 禁止：对闭集 `smallint` 字段只写“状态”“类型”“角色”“模式”而不写具体值域语义。
