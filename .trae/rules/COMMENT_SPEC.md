# 项目注释规范

适用范围：本仓库所有 `apps/*`、`libs/*`、`db/*`、`scripts/*` 下的 TypeScript / JavaScript 代码。

## 1. 核心原则

- 注释优先解释原因、约束、语义和风险，不逐句翻译代码。
- 先靠命名、拆函数、收敛类型提升可读性；只有代码本身不足以表达时才加注释。
- 注释必须随代码一起演进；失真注释比没有注释更糟。
- 默认使用简体中文，框架术语、库名、SQL/HTTP/Swagger/Drizzle/TSDoc 等专有名词保留英文。

## 2. 必须写注释的场景

- Service、Resolver、Extension、Helper 的**每个方法**（包括公共方法和私有方法）。
  - 类级注释说明整体职责，方法级注释说明前置条件、副作用、异常语义。
  - 私有方法重点说明：业务约束、校验规则、与其他方法的协作关系。
- `*.type.ts`、`*.types.ts` 中导出的稳定领域类型。
- 非直观事务语义、幂等语义、重试语义、补偿逻辑、事件出箱。
- 原生 SQL、复杂查询、性能取舍、兼容性取舍。
- 一次性迁移、repair、rebuild 脚本的执行前提与影响范围。

## 3. 可以不写注释的场景

- 命名已经足够清晰的简单 CRUD、getter、参数透传（**Service、Resolver、Extension 的方法除外**）。
- 与 Swagger、校验器、DTO、类型定义重复的说明。
- 单行映射或显然的控制流。

## 4. 分层要求

- Controller、DTO：保持克制，不重复 `ApiDoc`、`ApiPageDoc`、`ApiProperty` 或校验器已经表达的信息。
- Service、Resolver、Extension：重点说明前置条件、事务要求、副作用、兼容语义和失败语义。
- `db/schema`：只注释业务语义或特殊约束，不机械解释 `id`、`createdAt`、`updatedAt` 这类显然字段。
- Script / Seed：只在阶段划分真实反映依赖关系时使用阶段性注释。

## 5. 写法要求

- 导出的能力或类型优先使用 `/** */`；第一行写职责摘要。
- `@param`、`@returns`、`@throws` 只在不直观时补充。
- 行内 `//` 只用于说明局部约束、顺序依赖、兼容原因或关键分支。
- `TODO/FIXME` 必须写清楚原因和触发条件；临时兼容移除后同步删除。

## 6. 禁止写法

- 逐行翻译代码。
- 注释掉旧代码长期保留。
- 与实际行为不一致的历史描述。
- 重复“查询”“更新”“返回”这类无信息注释。
- 没有责任人、原因或清理条件的 `TODO/FIXME`。

## 7. 验收清单

- [ ] 注释解释的是原因、约束、语义或风险，而不是代码表面动作。
- [ ] 复杂能力、稳定领域类型、事务/幂等关键路径已有必要注释。
- [ ] Controller/DTO 没有堆积与 Swagger、校验器重复的注释。
- [ ] Schema 注释聚焦业务语义和特殊约束，没有机械字段说明。
- [ ] 不存在失真注释、注释掉的旧代码和无条件 `TODO/FIXME`。
