# 项目注释规范

适用范围：本仓库所有 `apps/*`、`libs/*`、`db/*`、`scripts/*` 下的 TypeScript/JavaScript 代码、Schema、扩展函数、Seed 与脚本。
本文件是仓库内默认注释规范来源。若注释与当前实现、类型契约、基础设施行为冲突，以当前可运行代码为准，并在修改时优先修正注释，而不是让错误注释继续扩散。

说明：

- Controller 规范参考 [.trae/rules/CONTROLLER_SPEC.md](D:/code/es/es-server/.trae/rules/CONTROLLER_SPEC.md)
- DTO 规范参考 [.trae/rules/DTO_SPEC.md](D:/code/es/es-server/.trae/rules/DTO_SPEC.md)
- Drizzle 规范参考 [.trae/rules/drizzle-guidelines.md](D:/code/es/es-server/.trae/rules/drizzle-guidelines.md)

## 1. 目标与原则

- **说明原因，不复述代码**：注释优先解释“为什么这样做”“约束是什么”“改动风险在哪”，不是把代码逐句翻译成人话。
- **分层一致**：不同层的注释密度不同。Controller/DTO 从简，Service/Resolver/Extension/复杂查询适度说明，Schema 只补业务语义与特殊约束。
- **面向维护者**：注释应帮助后来者快速理解跨模块契约、边界条件、事务语义、幂等语义与领域意图。
- **少而准**：优先通过命名、提取函数、类型收敛提升可读性；只有代码本身难以清楚表达时才加注释。
- **随代码一起演进**：代码变更导致注释失真时，必须同步更新或删除注释。

## 2. 范围与边界

### 2.1 覆盖范围

- 导出类、导出函数、导出类型、导出常量
- Service、Resolver、Extension、Query Helper、事务封装
- `db/schema` 中的表、关键字段、索引、约束
- Seed/脚本中的阶段性步骤说明

### 2.2 非目标

- 不用注释替代清晰命名
- 不用注释替代 Swagger、DTO 校验器或类型定义
- 不要求为每个文件、每个方法、每个字段机械补齐注释
- 不要求为了“注释统一”对存量代码做大范围重写

## 3. 总体规则

### 3.1 必须写注释的场景

以下场景必须补充注释，优先使用 `/** */`：

1. 导出的复杂能力：职责不止一眼可见的 Service、Resolver、Extension、Helper。
2. 稳定领域类型：`*.type.ts` 中每个导出类型都要说明用途与关键语义。
3. 非直观事务语义：需要透传 `tx`、依赖数据库错误码重试、幂等分支、补偿逻辑、事件出箱等。
4. 关键业务约束：例如“必须同源才能交换排序值”“只统计未删除记录”“这里不能走通用包装，否则会吞掉原始错误码”。
5. 无法从代码直接看出的性能或兼容性决策：批量查询、预加载、分页语义兼容、历史协议兼容。
6. 原生 SQL、复杂动态查询、魔法值映射：说明使用原因、输入边界与结果语义。

### 3.2 可以省略注释的场景

以下场景通常不写注释：

- 命名已经足够清晰的简单 CRUD 方法
- 单行字段映射、简单 getter、显然的参数透传
- 与 Swagger/`ApiProperty`/DTO 校验描述完全重复的说明
- “查询数据”“更新数据”“返回结果”这类没有新增信息的行内注释

### 3.3 注释写法要求

- 默认使用简体中文；框架术语、库名、SQL/HTTP/Swagger/Drizzle/TSDoc 等专有名词保留英文。
- 注释应直接说明约束、原因、语义，避免空话和修饰性语言。
- 优先写完整短句或清晰短语，不写口语化备注。
- 注释要与当前命名一致，使用仓库内统一术语，例如“应用用户”“评论目标”“软删除”“事务客户端”。
- 新增注释优先服务当前代码，不为历史实现背书。

## 4. 注释类型与格式

### 4.1 `/** */` 文档注释

适用场景：

- 导出类、导出函数、导出类型、导出常量
- 需要说明前置条件、异常语义、返回契约的公共方法
- 表、索引、关键字段、复杂工具函数

要求：

- 第一行先写一句职责摘要。
- 只有在参数、返回值、异常语义不直观时，才补 `@param`、`@returns`、`@throws`。
- 不要求为所有私有方法补全 TSDoc；私有方法仅在逻辑复杂或约束特殊时写。

### 4.2 `//` 行内注释

适用场景：

- 解释局部分支背后的原因
- 标记多步操作中的关键阶段
- 标识临时兼容、顺序依赖、锁/并发/幂等注意点

要求：

- 只注释当前代码块最关键的信息。
- 优先解释“为什么必须这样”，不要解释“这一行正在做什么”。
- 连续三行以上都需要解释时，优先抽函数或改为块级 TSDoc。

### 4.3 禁止的写法

- 用注释逐行翻译代码
- 注释掉旧代码长期保留
- 使用与实际行为不符的历史描述
- 在同一逻辑里重复写“查询”“更新”“返回”之类的无信息注释
- 没有完成条件、责任信息或清理时机的 `TODO/FIXME`

## 5. 分层规范

### 5.1 Controller

- Controller 不是注释重点区域。路由语义、Swagger 注解、DTO 命名已经清楚时，可以不写额外注释。
- 仅在以下情况补充简短注释：
  - 当前接口有明显的权限、租户、当前用户上下文语义
  - 路由动作与内部调用存在历史兼容原因
  - 需要提醒“这里只做编排，不要下沉数据库查询”
- 不要写与 `ApiDoc`/`ApiPageDoc` summary 重复的注释。

### 5.2 DTO

- DTO 注释优先由 `ApiProperty`、校验器与类型本身表达；普通字段通常不再写额外代码注释。
- 只有字段存在下列情况时才写注释：
  - 业务语义不直观
  - 单位、取值范围或枚举编码容易误解
  - 是历史兼容字段、快照字段、冗余字段
- `BaseXxxDto` 的注释口径必须与对应 Drizzle Table 一致，不能自造业务语义。

### 5.3 Service / Resolver

- 这是注释的主要落点之一。
- 类级注释说明职责即可，不写实现细节流水账。
- 方法级注释重点说明：
  - 前置条件与存在性约束
  - 事务要求与 `tx` 透传要求
  - 写路径的副作用、事件、计数、通知、缓存影响
  - 兼容性、幂等性、并发控制或失败回滚语义
- 简单查询方法如 `findById`、`getCount` 可只保留一句摘要，或在命名足够清楚时不写。

### 5.4 `db/schema`

- 表级注释可保留，用于说明实体职责。
- 字段级注释只在以下场景推荐：
  - 字段名无法直接表达业务语义
  - 字段是状态位、快照位、历史兼容位、配置位
  - 字段值编码、单位、来源、约束有额外语义
- 索引与唯一约束建议只给“为什么需要它”的说明，不必重复字段名。
- 新增或修改 schema 时，不要求为所有显然字段机械补注释，例如 `id`、`createdAt`、`updatedAt`。

### 5.5 Extension / Query Helper / Raw SQL

- 必须说明该 helper 解决什么问题，以及为什么不能直接靠普通 CRUD 表达。
- 算法型步骤允许用少量阶段注释，例如三步交换、先主查询再批量补充、先写后返回快照。
- 使用原生 SQL 时，注释需要说明：
  - 为什么不用 Drizzle 常规接口
  - 输入是否已参数化
  - 返回结构或排序语义的关键假设

### 5.6 Seed / Script

- 允许用阶段性注释组织流程，例如“基础配置数据”“业务基础数据”“核心业务数据”。
- 阶段注释应对应真实依赖关系，不要为了排版堆标题。
- 对一次性迁移或修复脚本，需写清楚执行前提与影响范围。

## 6. 推荐内容模板

### 6.1 类/函数摘要

- 说明职责边界，而不是实现步骤。
- 示例：`评论目标解析器，负责章节评论计数与评论权限校验。`

### 6.2 事务/副作用说明

- 说明必须带 `tx` 的原因、失败时的语义、是否依赖原始数据库错误。
- 示例：`这里保留原始 PostgreSQL 错误码，外层会基于冲突码做重试。`

### 6.3 领域类型说明

- 至少回答两个问题：
  - 这个类型服务哪个场景？
  - 哪些字段语义最关键？

### 6.4 行内关键注释

- 应聚焦约束而非动作。
- 好例子：`// 先写入临时值，避免唯一索引冲突。`
- 差例子：`// 更新第一条记录。`

## 7. TODO / FIXME 约定

- 仅在当前回合无法完成、但确实需要后续处理时使用。
- 必须写清楚原因与触发条件，避免留下无法执行的空备注。
- 推荐格式：

```ts
// TODO: 等评论举报统一到 interaction 域后，收敛这里的重复映射。
// FIXME: 这里暂时保留历史排序语义，前端仍依赖 1-based 回显。
```

- 临时兼容去除后，应同步删除对应注释。

## 8. 存量代码例外

当前仓库存在以下已知风格差异，新规范默认不要求立刻批量改造：

1. `db/schema/*` 中部分历史自动转换文件采用“表/字段/索引全量注释”风格。
2. `db/extensions/*` 中部分工具函数存在较高密度的步骤注释。
3. 个别 Resolver 方法使用了较完整的 `@param/@returns/@throws` 说明。

处理原则：

- 修改这些文件时，以“保留真正有信息量的注释、删除机械重复注释”为准。
- 不因为已有高密度注释，就在新代码里继续复制逐行解释风格。
- 若存量注释与当前行为冲突，优先修正注释；如短期无法统一，在变更说明中明确指出。

## 9. 验收清单

- [ ] 注释解释的是原因、约束、语义或风险，而不是逐句翻译代码。
- [ ] 导出的复杂能力、稳定领域类型、事务/幂等关键路径已有必要说明。
- [ ] Controller/DTO 没有堆积与 Swagger/校验器重复的注释。
- [ ] Schema 注释聚焦业务语义与特殊约束，没有机械复制显然字段说明。
- [ ] 行内注释数量克制，且主要用于关键分支、阶段步骤与兼容语义。
- [ ] 不存在失真注释、注释掉的旧代码、无条件的 TODO/FIXME。
- [ ] 变更代码时已同步修正相关注释，未让历史错误描述继续保留。
