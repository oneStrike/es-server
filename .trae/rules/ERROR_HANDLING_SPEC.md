# 错误处理规范

适用范围：本仓库所有 `apps/*`、`libs/*`、`db/*` 中通过 NestJS、Fastify、Drizzle ORM 暴露或传播的错误。
本文件是仓库内默认错误处理规范来源。若规范与当前可运行抽象、历史接口契约或重试/幂等语义冲突，以当前实现为准，并在变更说明中明确记录冲突点，而不是为了“形式统一”改坏现有行为。

## 1. 目标

- 对客户端返回稳定、可预期的错误状态与错误文案。
- 对服务端保留足够的原始上下文，便于定位约束名、表名、字段名、请求链路和业务场景。
- 让业务语义、数据库技术错误、全局兜底各司其职，避免重复映射与多处猜测。
- 对高并发、防重、重试场景优先使用幂等写法，而不是依赖异常驱动正常流程。

## 2. 总体原则

1. 业务错误在业务层定性，全局层只兜底，不代替业务层猜语义。
2. 数据库错误在 Drizzle 边界分类，不在 controller 或 filter 中重复发明第二套数据库语义。
3. 幂等冲突优先通过 `onConflictDoNothing`、`onConflictDoUpdate`、唯一业务键回查等模式消化，不把“重复提交”当成通用异常主路径。
4. 读接口上的附带写入必须可降级，不能因为浏览记录、阅读快照、埋点、通知等副作用失败拖垮主读链路。
5. 所有未知异常都必须被全局异常过滤器接住并落结构化日志，不能静默吞错。

## 3. 分层职责

### 3.1 ValidationPipe 层

- 只负责请求参数格式校验、DTO 转换与白名单过滤。
- 输出统一的 `400 Bad Request`。
- 不负责数据库、鉴权、权限、业务规则判断。

### 3.2 Controller 层

- 只做入参接收、用户上下文装配、调用 service。
- 不直接处理数据库异常，不在 controller 内做 `try/catch` 翻译数据库错误。
- 仅当接口层确实存在协议兼容需求时，才在 controller 侧补充异常转换。

### 3.3 Service / Resolver 层

- 业务规则失败统一抛出明确的 Nest 异常：
  - `BadRequestException`：业务规则不满足、状态不允许、参数语义非法
  - `NotFoundException`：资源不存在
  - `UnauthorizedException`：未登录或登录态无效
  - `ForbiddenException`：已登录但无权限
- 对用户可感知的唯一约束冲突，应在 service 就近翻译为明确文案，例如“邮箱已被使用”“无法重复关注”，不要直接把“数据已存在”暴露为最终业务语义。
- 对允许失败的非核心副作用，必须降级处理并输出结构化日志，不能静默吞错。

### 3.4 Drizzle 边界

- 所有常规写操作统一通过 `DrizzleService.withErrorHandling(...)` 或等价封装进入数据库。
- 需要保证资源存在的更新/删除，写后必须调用 `assertAffectedRows(...)`。
- `withErrorHandling(...)` 只负责把 PostgreSQL 错误分类成稳定的 HTTP 语义，不负责业务定制文案以外的复杂分支判断。
- 依赖原始 PostgreSQL 错误码做重试、幂等回退或冲突分支的路径，不要先经过会丢失原始错误类型的通用包装。

### 3.5 全局异常过滤器

- 负责统一错误响应结构。
- 负责记录结构化日志，包括请求信息与底层数据库上下文。
- 负责兜底未知异常，避免异常直接冒泡为未处理错误。
- 不负责维护第二套业务错误翻译规则。

## 4. 数据库错误处理规则

### 4.1 默认映射来源

- PostgreSQL 错误码、默认文案、默认状态码应以 `db/core/error/postgres-error.ts` 为单一来源。
- 全局异常过滤器不得再维护与其语义不一致的第二套数据库默认映射。

### 4.2 默认错误码语义

- `23505`：唯一约束冲突
- `23502`：非空约束冲突
- `23514`：检查约束冲突
- `40001`：事务序列化失败

### 4.3 业务翻译规则

- 若写操作对应明确业务含义，可在 `withErrorHandling(..., { duplicate: '...' })` 中覆盖默认文案。
- 若业务层需要区分不同唯一约束，应在 service 捕获后基于约束或场景重新抛出明确业务异常。
- 不要把数据库默认文案当作最终用户文案依赖。

### 4.4 原始上下文保留

- 数据库错误转换为 `HttpException` 时，必须保留原始 `cause`。
- 日志中必须能看到：
  - `errorCode`
  - `errorConstraint`
  - `errorTable`
  - `errorColumn`
  - `errorDetail`

## 5. 幂等、防重与重试

### 5.1 首选方式

- 首选数据库原生幂等写法：
  - `onConflictDoNothing`
  - `onConflictDoUpdate`
  - 稳定 `bizKey` + 写后回查

### 5.2 不推荐方式

- 不要依赖“先抛唯一约束异常，再根据异常 message 字符串判断是否重复”的模式。
- 不要把异常当作正常控制流主路径。

### 5.3 序列化冲突

- 对高并发事务冲突，允许基于 `40001` 做有限重试。
- 重试逻辑应基于原始错误码，而不是包装后的通用文案。

## 6. 读接口附带写入

- 浏览日志、阅读快照、埋点、消息通知、成长奖励等若不是主读结果的必要组成，必须设计为“失败可降级”。
- 详情页、列表页、聚合查询等 GET 接口不应因为这些附带写入失败直接返回 4xx/5xx。
- 若需要立即反映本次副作用结果，优先通过响应字段做本地推导，不强依赖副作用落库成功。

## 7. 日志要求

- 所有错误日志至少包含：
  - `requestId`
  - `path`
  - `method`
  - `ip`
  - `status`
  - `errorMessage`
- 数据库错误额外包含：
  - `errorCode`
  - `errorConstraint`
  - `errorTable`
  - `errorColumn`
  - `errorDetail`
- 幂等、防重、奖励、计数、通知等关键路径，建议额外包含：
  - `bizKey`
  - `targetType`
  - `targetId`
  - `userId`
  - `ruleType`
  - `costMs`

## 8. 当前仓库建议实现

1. `ValidationPipe` 继续负责参数校验，保留统一响应入口。
2. `TransformInterceptor` 继续负责成功响应包装，不介入错误语义。
3. `DrizzleService` 继续作为数据库错误分类与辅助断言的唯一入口。
4. `HttpExceptionFilter` 只负责统一错误响应和日志兜底，不再维护独立的数据库默认文案映射。
5. 业务 service 对“用户可理解的冲突”做就近翻译。
6. 幂等路径统一收敛到数据库幂等写法，减少基于异常 message 的回退逻辑。
7. 详情读接口上的浏览记录等副作用改为安全降级。

## 9. 优先改造清单

### P1

1. 收敛 `db/core/error/postgres-error.ts` 与全局异常过滤器中的 PostgreSQL 映射，保留单一来源。
2. 将作品详情、章节详情、论坛主题详情中的浏览日志写入改为安全降级，避免 GET 因附带写入失败而整体失败。
3. 清理依赖异常 message 魔法字符串判断重复的路径，优先改成 `onConflictDoNothing/Update + 回查`。

### P2

1. 完善全局 `ValidationPipe` 的嵌套 DTO / `children` 错误展开，避免复杂入参只返回模糊属性名。
2. 为高并发关键写路径补齐更稳定的结构化错误日志字段。
3. 对重复出现的唯一约束冲突，按约束名收敛为明确业务文案，而不是持续使用“数据已存在”。

### P3

1. 为错误处理增加集成测试，覆盖：
   - 参数校验错误
   - 资源不存在
   - 唯一约束冲突
   - 序列化冲突重试
   - GET 附带写入降级
2. 将常见幂等写法沉淀为仓库级 helper，减少业务模块重复实现。

## 10. 已知冲突点

当前仓库存在以下“规范与现状”冲突，后续收敛时应显式说明：

1. 部分唯一约束冲突仍沿用默认文案“数据已存在”，尚未按约束名完全收敛为明确业务语义。
2. 错误处理相关集成测试仍不充分，尤其缺少 GET 附带写入降级与事务序列化冲突重试覆盖。
